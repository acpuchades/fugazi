# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`fugazi` is a Rust library (edition 2024, minimal dependencies) of **incremental** technical-analysis primitives. Every primitive owns its internal state and is advanced one sample at a time via `update()`, carrying just enough intermediate state to produce the next output in ~O(1). The same code therefore serves both live streaming and batch backtesting.

Unconditional deps: `serde` + `serde_json` (sources module + CLI YAML/JSON), `time` (calendar `Timestamp` decomposition), `statrs` (standard-normal CDF/Φ⁻¹ for PSR/DSR). Everything else is behind three default-on features: **`sources`** (remote providers, async HTTP), **`runtime`** (pure type-erasure vocabulary in [`fugazi::runtime`](#cli-internals-srccli)), **`cli`** (the binary; implies both). A new unconditional dep is still a judgment call — reach for a stateless closed-form primitive first.

### Bumping the package version

The version is duplicated in **four** places that must stay in lockstep — a `cargo build` catches a Rust/Rust drift but not a Python or README one, so it's worth a checklist:

1. **`Cargo.toml`** — `[package] version = "X.Y.Z"` (the workspace root).
2. **`python/Cargo.toml`** — `[package] version = "X.Y.Z"` (the pyo3 cdylib crate).
3. **`python/pyproject.toml`** — `[project] version = "X.Y.Z"` (the wheel metadata; this is the version `pip install fugazi` sees).
4. **`README.md`** — the `[dependencies]` snippet under `## Install` (`fugazi = "X.Y"` — track the major.minor there, not the patch).

Then `cargo build --workspace` (updates `Cargo.lock`), commit all five files (`Cargo.toml`, `python/Cargo.toml`, `python/pyproject.toml`, `README.md`, `Cargo.lock`), tag `vX.Y.Z`, push `main` and the tag. `python/README.md` intentionally has no version string.

## Architecture

Three composable layers: indicators (numeric sources), signals (boolean-valued indicators — `Indicator<Output = bool>`), and strategies (the decision layer that trades into a wallet).

### Indicators = the numeric *sources* (`src/indicator.rs`, `src/indicators/`)

`Indicator` has associated `Input`/`Output`, `update(&mut self, Input) -> Option<Output>`, `value()`, `is_ready()`, `reset()`, plus two warm-up introspection methods:

- **`warm_up_period()`** (required) — the *exact* number of samples before the first `Some`, accounting for the whole composed chain. Wrappers add their window on top of `source.warm_up_period()`; binary carriers take the max of both sides. `tests/warm_up.rs` asserts exactness across the catalogue — add a new indicator to that battery.
- **`unstable_period()`** (default `0`) — extra samples IIR smoothers need for the seed's residual weight to decay below `SETTLE_TOLERANCE = 1e-3` (in `smoothing.rs`). `EmaState`/`WilderState` expose their own `unstable_period()` under the same name; wrappers sum it into `source.unstable_period()`.
- Provided **`stable_period()`** = warm-up + unstable.

Output is `Option` because most indicators need a warm-up (`None` until ready).

The defining design choice: **price-series indicators own their input source** and are generic over it — `Ema<S>`, `Sma<S>`, `Rma<S>`, `Rsi<S>`, `Macd<S>` where `S: Indicator<Output = Real>`, with `Input = S::Input`. So composition is just nesting constructors:

```rust
Ema::new(Current::close(), 20)          // EMA-20 of the close (Input = Candle)
Ema::new(Sma::new(src, 10), 20)         // EMA of an SMA
Rsi::new(Identity::new(), 14)           // RSI of a raw Real stream
```

There is **no pipe/`then`/`Chain`** — chaining *is* construction.

- **Leaf sources** terminate the chain: `Value<I>` (constant, ignores input), `Identity<I = Real>` (input-typed passthrough; `Identity::<Atom>::new()` is the *default source* for the candle/calendar leaves below), and the `Current` candle accessors (`Current::close()`, `Current::volume()`, …; built on `Field<F, S>` / `CandleField` in `candle.rs`). Every source-generic leaf — `Field<F, S>` (`Close`/`High`/`Low`/`Open`/`Volume`/`Typical`/`Median`), `CurrentBar<S>`, `Calendar<F, S>` (`Year`/`Month`/`DayOfWeek`/…), `CurrentTime<S>`, `IsWeekday<S>`, `IsWeekend<S>` — is generic over an atom-emitting source `S: Indicator<Output = Atom>`, defaulting to `Identity<Atom>`. Convention: `T::new()` uses the default source, `T::of(source)` re-roots on a custom one (a `Pick` for cross-asset frames, an atom-side `Latch`/`Resample`, …). Type aliases (`Close`, `Year`, `IsWeekday`, …) carry `<S = Identity<Atom>>` so `Close::<CustomSource>::of(...)` type-checks.
- **Cross-asset sources** (`time.rs` + `snapshot.rs` + `indicators/pick.rs`) let a chain reach into a multi-asset input frame. Three core types (re-exported at the crate root, through `prelude`, and via `fugazi::types`):
  - **`Frequency`** (`src/time.rs`) — bar cadence enum (`Minute(u32)` / `Hour(u32)` / `Day(u32)` / `Week(u32)` / `Month(u32)`), totally ordered **by duration**, `FromStr` on `N<unit>` alphabet (`m`/`h`/`d`/`w`/`M`). Canonical bar-cadence type; `cli::calendar::Frequency` re-exports it; `sources::Interval` is its provider-side twin.
  - **`Selector { symbol: Option<String>, freq: Option<Frequency> }`** — partial key naming which asset a `Pick` reads. `None` fields wildcard-match any storage value; `Some` must equal. `Selector::default()` = empty (single-entry-unpack path); shorthands `by_symbol` / `by_freq` / `exact`.
  - **`Snapshot<K>`** (`src/snapshot.rs`) — newtype around `HashMap<K, Atom>` with `get`/`insert`/`iter`/`FromIterator`/`From<HashMap>` + `sole_atom(&self) -> Option<&Atom>` (returns unique atom in size-1, panics on 2+, `None` on empty). `impl Snapshot<Selector>` adds `find(query)` for wildcard lookup. CLI/Python use `Snapshot<Selector>` only.

  **`Pick<S = Identity<Snapshot<Selector>>>`** (no `K` generic — `Selector` is baked in) projects one asset out: `Input = S::Input`, `Output = Atom`. Constructors: `Pick::new()` (empty selector, sole-atom unpack), `Pick::matching(selector)` (structural match; empty selector also dispatches to `sole_atom`), `Pick::of(selector, source)` for a custom snapshot source. Composes with every source-generic leaf: `Close::of(Pick::matching(Selector::by_symbol("BTC")))` is BTC's close; `.sub(Close::of(Pick::matching(...)))` is the BTC/ETH spread; `Atr::new(CurrentBar::of(Pick::matching(...)), 14)` is BTC's ATR. A non-empty selector fires `None` when the query matches no entry; an empty selector **panics** on a 2+ snapshot (the "single-series strategy accidentally hooked to multi-asset input" tripwire; surfaces to Python as `RuntimeError`). **`Atom` equality is by `time`** — atoms compare equal iff their bar-open `Timestamp`s match; `PartialOrd`/`Ord` sort chronologically with `None` first. So snapshots dedup by time and mixed atom slices sort into run order without a custom key.

  Python mirror: `ta.Frequency("1h")`, `ta.Selector(symbol="BTC", freq="1h")`, `ta.Snapshot({...})`, `ta.pick(symbol=..., freq=...)`. Snapshot dict-key coercion accepts `str` / `Frequency` / `(str, freq)` / `Selector`. `ta.pick()` (no args) inherits sole-atom-or-panic semantics. The YAML `!pick { symbol, freq }` tag and "root-ish indicators default to `!pick`" are the last remaining piece for `fugazi-cli`.
- **Position-anchored sources** (`indicators/position.rs`) read the live position rather than the candle stream. A strategy owns one shared `Position` (`Rc<RefCell<PositionState>>`) holding its signed `size`, `entry` price, and the `peak`/`trough` since entry; it updates it from its fills (`apply(side, units, price)`) and each bar (`update(candle)`, folding completed-bars extremes). Its accessors `position.entry()` / `.peak()` / `.trough()` each return a `PositionField` — the field projected into an `Indicator<Output = Real>` (`None` while flat) — so a stop-loss / take-profit **level** is an expression: `position.entry().sub(Atr::new(14).mul(...))` for a fixed stop, `position.peak().mul(Value::new(0.95))` for a trailing one. Only meaningful inside a strategy that drives the `Position`.
- **Bar indicators** consume the whole `Candle` directly (they are not "of a price"): `Atr`, `Adx`, `TrueRange`, the range-based volatility estimators `Parkinson`/`GarmanKlass`/`RogersSatchell`, and the volume indicators `Obv`, `Vwap`, `Ad` (Chaikin A/D line), `Mfi` (money-flow index). These take only parameters, or none, e.g. `Atr::new(14)`, `Obv::new()`, `Mfi::new(14)`. The cumulative ones (`Obv`/`Vwap`/`Ad`) anchor at construction; `reset()` re-anchors — e.g. at a session boundary for `Vwap`. The three volatility estimators roll a mean of a per-bar OHLC estimator over `period` bars (backed by the shared `WindowStats` core, so O(1)/bar and `unstable_period() = 0`), then take the square root — more efficient than a close-to-close `StdDev` because they use the full intraday range: `Parkinson::new(source, period)` uses the high/low range only (`sqrt((1/(4·ln2))·mean(ln(H/L)²))`); `GarmanKlass` adds the open/close move (`mean(0.5·ln(H/L)² − (2ln2−1)·ln(C/O)²)`, mean clamped ≥0 before the root since its per-bar term can go slightly negative); `RogersSatchell` is drift-independent (`mean(ln(H/C)·ln(H/O) + ln(L/C)·ln(L/O))`, non-negative by construction). Yang-Zhang is deliberately absent — its overnight-gap term is meaningless on 24/7 crypto. YAML: `!parkinson`/`!garman_klass`/`!rogers_satchell { source?, period }`; Python: `ta.parkinson(period)` / `ta.garman_klass(period)` / `ta.rogers_satchell(period)`.
- **Two-source indicators**: `Donchian<H, L>` takes a high source and a low source, e.g. `Donchian::new(Current::high(), Current::low(), 20)`.
- **Windowed statistical estimators** over a Real source: `Skewness`/`Kurtosis` (standardized 3rd/4th central moments; kurtosis raw, ~3 for normal), `ZScore` (`(x − SMA) / stddev`), and the two-source `Correlation` (rolling Pearson; autocorrelation via `Correlation::new(x.clone(), x.lag(n), period)`) are all O(1)/bar off the shared `WindowStats`/`WindowCovariance` cores (`unstable_period() = 0`, dispersion-free window reads `0.0`). YAML: `!skewness`/`!kurtosis`/`!zscore { source?, period }`, `!correlation { lhs, rhs, period }`; Python `ta.skewness`/`ta.kurtosis`/`ta.zscore`/`ta.correlation`. **`VarianceRatio` is the deliberate exception to the crate's O(1) promise** (`indicators/variance_ratio.rs`): a Lo-MacKinlay variance-ratio regime classifier over the source's *first differences* — reads `1.0` under the random-walk null, **`> 1` trending** (positively autocorrelated), **`< 1` mean-reverting**. Lo-MacKinlay contrasts overlapping `lag`-period return variance against one-period variance across the whole window, which does **not** reduce to an incremental recurrence, so it retains the last `period` samples in a `VecDeque` and recomputes from scratch on every full-window update — **O(`period`)/bar, not O(1)**. That is fine for the intended use (an offline overlay feature column: a regime score joined against a strategy's windowed metrics) and acceptable for moderate windows live; the cost is documented on the type and called out here because it breaks the incremental convention the rest of the catalogue holds to. Compose the log yourself for log-return VR (`VarianceRatio::new(Log::natural(Current::close()), 60, 5)`); a dispersion-free window (constant one-period returns) reads the `1.0` null. `new` asserts `lag ≥ 2` and `period ≥ lag + 2` (the unbiased overlapping estimator needs >1 block); `warm_up_period()` stays exact and is in the `tests/warm_up.rs` battery. YAML: `!variance_ratio { source?, period, lag }`; Python `ta.variance_ratio(source, period, lag)` (the two constraints surface as `ValueError`, not a Rust panic).
- **Calendar sources** (`indicators/calendar.rs`) decompose `atom.time` (`Option<Timestamp>`, UTC ms): `Year`, `Month`, `Day`, `Hour`, `Minute`, `Second`, `DayOfWeek` (ISO 1=Mon..7=Sun), `DayOfYear`, `WeekOfYear` (ISO), `Quarter`, `UnixSeconds`, `UnixMillis` — all on the `Calendar<F, S> + CalendarField` pattern (same shape as `Field<F, S> + CandleField`, `S: Indicator<Output = Atom>` default `Identity<Atom>`). Plus the `CurrentTime` leaf (raw `Timestamp`) and bool signals `IsWeekday` / `IsWeekend`. CSV loader (`src/cli/data.rs`) populates `Atom::time` at load; every remote provider returns `Vec<Atom>` with time already set — so calendar accessors fire on every CSV-fed or remotely-fetched bar. `atom.time` is only `None` for synthetic atoms (`Atom::new(candle)`) or unparseable time labels; accessors emit `None` and the two signals read `false` in that case. Daily-and-higher bars conventionally sit at 00:00 UTC, so `Hour`/`Minute`/`Second` are identically `0.0` and only day-and-above accessors are meaningful. Composition: "Monday" = `!eq { lhs: !day_of_week, rhs: !value 1 }`, "morning window" = `!lt { lhs: !hour, rhs: !value 9 }`. YAML: bare `!year` / `!month` / `!day` / `!hour` / `!minute` / `!second` / `!day_of_week` / `!day_of_year` / `!week_of_year` / `!quarter` / `!unix_seconds` / `!unix_millis` / `!time` / `!is_weekday` / `!is_weekend`. Python: `ta.year()` … `ta.unix_millis()`, `ta.is_weekday()` / `ta.is_weekend()`; `ta.Atom(candle, time=<ms>)` and `.time` getter. Uses the `time` crate — hence the unconditional dep.
- **`Unstable<S>`** (`indicators/unstable.rs`) is a **transparent passthrough** wrapper forwarding every method to `S` *except* `unstable_period()`, which it forces to `0` (`warm_up_period()` untouched). Built with `Unstable::new(source)`; fluent `.unstable()` on `IndicatorExt` and `BoolIndicatorExt`. YAML: `!unstable { source }` / `!unstable { signal }`. This is the **opt-in override** for the readiness gate — a caller who's happy to trade through a subtree's IIR settling tail wraps it so the strategy's `is_ready()` only waits for warm-up. See [Safe defaults, opt-in overrides](#safe-defaults-opt-in-overrides).
- `Real = f64` and `Candle` (OHLCV) live in `src/market.rs` alongside the `Atom`/`OverlayInfo`/`Schema` machinery (the market-data vocabulary the indicator chain consumes). `src/types.rs` is now a thin **facade** re-exporting the split modules (`time` / `market` / `snapshot`), so `fugazi::types::*` import paths still resolve unchanged; new code can use the direct module paths.
- Multi-output indicators (`Macd`, `Adx`, `Bollinger`, `Donchian`, `Keltner`, `Aroon`, `Dmi`) expose one named field per output and set `Output` to a `Copy` struct (`MacdValue`, `AdxValue`, …); single-output ones expose `value: Option<Real>`. Each has a **component accessor per output** (`macd.line()`/`.signal()`/`.histogram()`, `bands.upper()`/`.middle()`/`.lower()`, `adx.adx()`, `dmi.plus_di()`, …) returning a `Component<Self>` — the field projected back into an `Indicator<Output = Real>`, so it composes and compares like any other source: `macd.line().crosses_above(macd.signal())`, `Current::close().gt(bands.upper())`. Each accessor **clones** the source (one independently-advanced instance per component). The `Component` carrier (`indicators/component.rs`) holds the source plus a `fn(Output) -> Real` selector. **Accessor bodies are generated by the `component_accessors!` macro** (e.g. `component_accessors!(Bollinger<S>, BollingerValue; upper => upper, middle => middle, lower => lower);`) — invoke it once per new multi-output indicator; don't hand-write the bodies.
- `StochRsi<S>` is a type alias for `Stochastic<Rsi<S>>` — StochRSI is just the stochastic transform over an RSI source: `Stochastic::new(Rsi::new(src, 14), 14)`.

### Signals = boolean-valued *indicators* (`src/signal.rs`, `src/indicators/{compare,logic,ext}.rs`)

A **signal is just an `Indicator<Output = bool>`** — there is no second trait hierarchy. `Signal` is a thin marker, `trait Signal: Indicator<Input = Candle, Output = bool>` (blanket-impl'd, `?Sized`), naming "a boolean condition over a `Candle`" so a strategy can hold one as `Box<dyn Signal>`. Like any indicator a signal is `None` until warmed; read it as a plain `bool` (false until ready) with `BoolIndicatorExt::is_true` (= `value().unwrap_or(false)`). There is no `src/signals/` module — the comparison/logic/ext pieces live under `indicators/` and are imported from there.

- **Comparisons are bool-output binary ops** over two real sources: aliases `Gt`/`Lt`/`Ge`/`Le`/`Eq`/`Ne` for `Combine<L, R, GtOp>` etc. (`indicators/compare.rs`). The op is **value-carrying** — each holds an absolute `epsilon` (default `DEFAULT_EPSILON = 1e-8`); the fluent `.gt()`/`.lt()`/… builders use the default, `Gt::with_epsilon(a, b, eps)` overrides.
- **Boolean logic** (`indicators/logic.rs`): `And`/`Or`/`Xor` are `Combine<L, R, AndOp>` etc. over two bool sources; `Not` and `Change` are dedicated unary bool-output carriers; `Const<In>` is the constant-bool leaf (the twin of `Value`).
- `IndicatorExt` (blanket over every `Real`-output indicator, in `indicators/ext.rs`) is the fluent builder for **operators only** — comparisons (`gt`/`lt`/`ge`/`le`/`eq_to`/`ne_to`, `above`/`below` — note `eq_to`/`ne_to`, not `eq`/`ne`, to avoid collision with `PartialEq`), arithmetic (`add`/`sub`/`mul`/`div`), lookback (`lag`/`diff`/`ratio`/`roc`), rolling extremum (`rolling_max`/`rolling_min`), the unstable-tail passthrough (`unstable`), and the composed `crosses_above`/`crosses_below`. Named indicators (`Sma`, `Bollinger`, …) are **not** builder methods; construct via their own `::new`. Do not add `.sma()`/`.bollinger()`-style builders.
- `BoolIndicatorExt` (blanket over every `Indicator<Output = bool>`, `?Sized` so it works on `Box<dyn Signal>`) adds the bool view `is_true()` and the combinators `and`/`or`/`xor`/`not` + the single edge primitive `changed` (a `Change` toggle detector) + the unstable-tail passthrough `unstable` (an `Unstable<Self>` that reports `unstable_period() = 0` while forwarding the signal's output).
- **A crossover is not a primitive**: `crosses_above(a,b)` expands to `a.gt(b).and(a.gt(b).changed())` — "comparison is true *and* it just changed". (Clones the operands, so ~2× the source work.)

### Strategies = the decision layer (`src/strategy.rs` for the `Strategy` trait; `src/wallet.rs` for the wallet vocabulary)

Unlike the pure layers below it, a strategy **acts**. The `Strategy` trait has:

- `update(&mut self, Input)` — advance signals/indicators. `&mut self` only, so updates across strategies are independent and parallelizable.
- `on_fill(&mut self, &Order<Symbol>)` — hook the driver calls for each fill the wallet books; lets a strategy track its own position from the fill stream. Default no-op.
- `is_ready(&self) -> bool` — safe-by-default readiness gate. Driver skips `trade()` while `false` but `update()` / `on_fill()` still run. Default `true`.
- `trade(&self, &mut dyn Wallet<Symbol>)` — read state and open/adjust/close positions. `&self`, price-free. Serial against a shared wallet (sizing resolves against running state).
- `reset()` — associated `Input` / `Symbol`.

Each bar the driver: feeds each symbol to the wallet (`update` returns that tick's fills), routes each fill to every strategy's `on_fill`, `update`s each strategy, then `trade`s each *only if* `is_ready()` returns `true`. There is deliberately **no one-shot `evaluate`**.

**`SingleAssetStrategy<Sym>`** (`strategies/single_asset.rs`, `Sym: Clone + PartialEq`) is the concrete `Strategy` (Input = Candle) covering the classical shape — long/flat/short driven by a few boolean signals, sized against equity via a real-valued multiplier. Four `Box<dyn Signal>` slots (open/close long, open/close short), each defaulting to `Const::<Candle>::new(false)`; unwired protective levels are `None`.

- **Readiness gate.** Tracks `bars_seen: usize` and overrides `is_ready()` = `bars_seen >= max(stable_period())` across every wired signal, protective level, and the sizing indicator. Wrap a subtree in `Unstable` to contribute `0`. Purely windowed (FIR) chains have `unstable_period() = 0` throughout, so the gate elapses on the last warm-up bar with no override needed.
- **Builders.** `long_on(enter, exit)`, `short_on(enter, exit)` (chainable; opposite-side entry reverses), and the `buy_and_hold(symbol)` constructor.
- **Protective levels.** Per-side stop-loss / take-profit as `Box<dyn Indicator<Output = Real>>` via `long_stop_loss` / `long_take_profit` / `short_*`, built against the strategy's own `Position` (via `position()`). Examples: `position.entry().mul(Value::new(0.95))` (fixed 5% long stop), `position.peak().mul(Value::new(0.95))` (5% trailing), `position.entry().sub(Atr::new(14).mul(Value::new(2.0)))` (ATR stop).
- **Position tracking.** Reads the wallet only through the fill stream: `on_fill` calls `position.apply` setting entry at the real fill price and clearing on flatten. Wallet stays a pure execution venue.
- **`trade` sequence.** Read sizing → skip whole call on `None` (safe-by-default). Entries first (sizing-scaled, reversal-capable) → flatten-to-flat signal exits → rest protective levels on the active side.
- **Order semantics.** Entries and signal exits are **market orders** (`set` / `close` then `cancel_protective`), queued and filled a bar later at next `open` — signal bar never fills. Protective stops are **resting orders the wallet owns**: strategy re-submits the level each bar as `set_stop` / `set_take_profit` (idempotent, latest-wins, so trailing levels cancel/replace); the wallet triggers and prices the fill (at level, or bar's `open` on a gap). Trailing levels track *completed* bars (`Position::peak`/`trough` exclude the bar in progress), so a resting stop fills one bar later than a fixed stop would.
- **Not a rule engine.** `SingleAssetStrategy` is a concrete parameterised type, not a policy-object trait. Don't add `(signal, action)` tables or a `RuleStrategy`-style layer without being asked.

The **`src/strategies/` catalogue** is free-function specializations returning `SingleAssetStrategy` (`ma_crossover`, `rsi_reversal`, `donchian_breakout`, `keltner_breakout`, … under `trend` / `mean_reversion` / `momentum` / `volume` / `composite`): long/flat = `new(sym).long_on(enter, exit)`, always-in = `new(sym).long_on(up, down).short_on(down, up)`. `ZScoreReversion` doesn't fit the mould (reads its `z` indicator directly; long/short with a flat rest) and stays its own bespoke `Strategy`, reading the wallet's position via private `is_long`/`is_short` helpers at the top of `strategies/mean_reversion.rs`.

**Sizing.** Positions size via `value_frac(m)` where `m` = the current value of the **position-sizing indicator** (set via `position_sizing(...)`, default `Value::new(1.0)`). `m` is magnitude only (direction stays with the entry's `Side`), read on transitions only, and folded into the readiness gate. `Value::new(0.5)` = fixed half-position; a Kelly indicator = Kelly sizing; `target_vol.div(realized_vol)` = vol targeting. Since `value_frac` sizes against equity (surviving a reversal), one `set` reverses and re-sizes exactly — no `enter_all_in` helper, no scale-in/scale-out primitives. Continuous mid-position rebalancing isn't in scope.

<a id="book-anchor"></a>**Book anchor.** Alongside `Position` (current open trade), `SingleAssetStrategy` owns a shared `Book` (`Rc<RefCell<BookState>>`, `src/indicators/book.rs`) tracking strategy-lifetime state — cash, position units, marked-to-market equity, running peak, per-bar returns, and closed-trade summaries staged pending → active so per-close accessors emit `Some` only on the closing bar. Six `BookField` accessors: `book.equity()` / `.equity_peak()` / `.drawdown()` / `.return_per_bar()` / `.trade_pnl()` / `.trade_return()` — ordinary real indicators driving the book-anchored sizing recipes. A rolling `Sma`/`StdDev` over `book.trade_return()` naturally reads "over the last N closed trades" because those cores only advance on `Some`. Seeds at `initial_equity`; match to the wallet's starting cash for meaningful numbers. `SingleAssetStrategy::new(sym)` defaults to `1.0` (toy setup); `with_initial_equity(sym, cash)` is what the CLI's `--cash` flag threads through.

`PairsStrategy` owns a `Book<Sym>` on the same terms (`with_initial_equity(left, right, cash)`, `new(...)` defaults to `1.0`). Both legs' fills feed one cash balance; both legs' closes mark to market each bar; a trade closes when both legs are flat; per-trade P&L sums both legs' contributions. So book-anchored sizing recipes work on a pair the same as on a single asset. `Book` is generic over `Sym` internally (`HashMap<Sym, LegState>`); single-asset registers one leg, pairs registers two. Cross-zero on any leg closes+opens the aggregate trade (rare on pairs since `PairsStrategy` always flattens before flipping).

**`BasketStrategy<Sym>`** (`strategies/basket.rs`) is the N-symbol cross-sectional twin. Reads the whole `Snapshot<Sym>` each bar: scores every symbol with a per-symbol *scoring source*, applies a **selection closure** (`Box<dyn Fn(&HashMap<Sym, Real>) -> HashMap<Sym, Side>>`) to decide long/short/flat, and drives the wallet.

- **Selection.** Ready-made: `strategies::basket::{top_bottom, threshold, quantile}` and wrapping builders `.top_bottom(longs, shorts)` / `.threshold(long_min, short_max)` / `.quantile(long_q, short_q)`. `.selection(closure)` is the escape hatch (signal-gated rank, stateful selector reading `book()`, …).
- **Floating universe.** Symbols aren't declared upfront; two factories (`Fn(&Sym) -> impl Indicator<Input=Snapshot<Sym>, Output=Real> + 'static`) are called once per symbol on first sight, expected to root leaves on the current symbol via `Pick::matching(Selector::by_symbol(sym.clone()))`. Absent symbols keep their chain but roll off `latest_score` as `Pick` reads `None`.
- **Sizing.** Per-leg `ValueFraction` magnitude, **no auto-normalization** — `sized_by(|_| equal_weight(N)).top_bottom(N/2, N/2)` gives 100% gross; any deviation is an explicit literal. `sizing::equal_weight(n_legs) = Value::new(1.0 / n_legs)` is the common-case helper. Price-based (`vol_target`, `atr_risk`) and book-anchored (`drawdown_throttle`, `equity_vol_target`, `fractional_kelly`) recipes work per-symbol from inside the factory.
- **Costs** stay on the wallet — `PaperWallet::set_costs_for(sym, ...)` per-symbol, transparent to every fill.
- **Per-symbol readiness.** `is_ready()` returns `true` unconditionally; the gate is enforced inside `trade()` by only ranking symbols whose score chain has read `Some` this bar. A whole-strategy gate would refuse to trade until every symbol in a floating universe warmed up.
- **State.** Per-symbol `Position` trackers (for per-leg protective levels, not wired yet) + one shared `Book<Sym>` on the same terms as `PairsStrategy`. Seed with `BasketStrategy::with_initial_equity(cash)`; `new()` defaults to `1.0`.
- **Transitions** are market orders (`wallet.set` / `wallet.close` then `cancel_protective`), only when target side differs from current position — no per-bar rebalance churn on unchanged selections.
- **Not shipped this pass**: `.dollar_neutral()`, `.rebalance_every(...)`, per-leg protective levels, YAML `!basket` builder with `SpecTemplate<ExprSpec>`-backed score/sizing, Python bindings (Strategy-layer bindings are intentionally out — see parity discipline).

`src/strategy.rs` carries only the `Strategy` trait (~80 lines). The `Wallet` vocabulary — trait, in-memory impl, shape types (`Order`/`Ack`/`OrderId`/`OrderKind`/`Side`/`Size`/`Reference`/`Units`/`WalletError`) — lives in **`src/wallet.rs`** so a downstream broker crate implementing `Wallet` doesn't drag in the `Strategy` machinery. Crate-root and prelude re-exports (`fugazi::PaperWallet`, `fugazi::Wallet`, `fugazi::prelude::*`) still expose everything. Everything below is in **`src/wallet.rs`** unless noted:
- **`Wallet<Sym>` is a trait** (`&mut dyn`) — the single **seam** between pure fugazi and a downstream execution system. fugazi ships only the in-memory paper impl; a downstream crate implements `Wallet` with a type whose `set_position` publishes to an event bus / broker. The wallet is **priced from outside**: `update(symbol, candle) -> Vec<Order>` feeds each symbol's bar every tick (the `close` marks to market, the `[low, high]` bounds fills), and returns the fills booked on that tick (the stream the driver routes to `on_fill`). Query with `funds()` / `position(&Sym)` / `price(&Sym)` / `equity()`.
  - **Submitting is not filling.** Market moves — `set_position` (absolute units), `set` (`Side` + `Size`, absolute target; opposite side reverses), `close` — return `Ack` synchronously: `Ack::Filled(Order)` on-the-spot or `Ack::Working(OrderId)` when the fill arrives later in the stream. Resting protective legs — `set_stop(sym, trigger)` / `set_take_profit(sym, trigger)` (idempotent, latest-wins per symbol; the wallet reads the side from the position) and `cancel_protective(&sym)` — are triggered and priced by the wallet.
  - **`PaperWallet` timing.** Queues market moves and fills at the *next* bar's `open`; protective legs fill when a bar trades through the trigger (at the level, or `open` on a gap). So a backtest never fills on the signal's own bar. A live impl fills on the venue's schedule.
  - **Errors.** `WalletError` (`UnknownPrice`, `InvalidPrice`, `PriceOutOfRange`, `InsufficientFunds`) is returned by a live movement that can't be accepted; on `PaperWallet` an infeasible queued fill is silently dropped, so its public methods return `Ok`.
  - **No explicit-price primitive on the trait** — fill-pricing lives in the impl. **No `trade(delta)`** — scale-in is `set_position(position + delta)`. NB: the trading/event-bus system itself is not in fugazi; keep market/IO code out of this crate.
- **Unit-tagged amounts** keep reference currency and instrument units from mixing: `Reference(Real)` (quote/funds — `funds`/`equity`/`price`) and `Units<Sym> { symbol, amount }` (signed instrument units — `position`, `set_position`). `Order<Sym>` (`{ symbol, side, units, price, kind, id }`) stays plain `Real` — `symbol`+`side` imply the unit; carries the fill `price`, an `OrderKind` (`Market`/`Stop`/`TakeProfit`, distinguishing market from trigger fills), and the `OrderId` of the submission it fills (correlating to `Ack::Working`). `Order::from_delta(symbol, delta, price, kind, id)` builds the buy/sell for a position change (`None` within `DEFAULT_EPSILON`).
- **`PaperWallet<Sym>`** — pure in-memory `Wallet` impl: `funds` + `HashMap<Sym,Real>` positions + `HashMap<Sym,Candle>` bar map + `HashMap<Sym,Pending>` queue of pending market orders + `HashMap<Sym,Protective>` resting brackets + a `Vec<Order>` blotter + `next_id` counter. Market `set_position`/`set`/`close` queue one move per symbol (latest wins) and return `Ack::Working`; `set_stop`/`set_take_profit` register one bracket per symbol (latest wins); `cancel_protective` drops it. `update(symbol, candle)` marks the bar, flushes the queued market order at `open`, then matches the resting legs against `[low, high]` (stop-loss precedence; a fill flattens and OCO-cancels the whole bracket, as does a market exit/reversal). All fills funnel through the private `fill_at`; the resting-order fill price is provably in `[low, high]`, so `PriceOutOfRange` is unreachable on the paper path. Adds inherent `new`, `positions()`, `orders()`, `reset()` (restores to freshly-constructed state; seed `funds` retained on the struct).
- **`Ack<Sym>`** (`Filled(Order)` | `Working(OrderId)`) is the synchronous submit result; **`OrderId(u64)`** is wallet-minted so a later fill correlates to its submission. Execution stays **synchronous** — the per-bar loop is the event loop; live fills arriving between bars reach the strategy via `on_fill`.
- **`Size`** (the magnitude vocabulary): `Units(n)` absolute, `FundsFraction(f)` (= `f·funds/price`, cash), `ValueFraction(f)` (= `f·equity/price`, all-in/target-weight; `1.0` flips cleanly on a reversal), `PositionFraction(f)` (= `f·|position|`, adjust-only). `resolve(price, position, funds, equity) -> magnitude`. Direction comes from `Side` (`Buy`/`Sell`, `.sign()`), not the size.
- There is **no `Market` trait**: the wallet holds its own fed prices, so a multi-asset input just feeds several symbols via `update` and a strategy's `trade` acts on several symbols in one call (multi-asset/pairs in the same type).
- Sizing/direction/short-selling/always-in-market are all just *what the strategy's code does* — no flags. Python (`python/src/lib.rs`) binds `PaperWallet`/`Order`/`Size` (sides as `"buy"`/`"sell"` strings, symbols as `str`; `update` returns the fills list, `set`/`set_position`/`close`/`set_stop`/`set_take_profit`/`cancel_protective`, `Order.kind` as `"market"`/`"stop"`/`"take_profit"`, `WalletError` → `ValueError`); a Python "strategy" is plain Python code driving a `PaperWallet`.

### Run driver = the pure per-bar loop (`src/backtest.rs`)

**`fugazi::backtest::run(&mut strategy, &mut wallet, snapshots) -> RunReport<Sym>`** is the pure primitive walking a `Strategy` over a snapshot stream through any `impl Wallet<Sym>` — not backtest-only (a live-`Wallet` impl works too), hence the neutral name.

**Per-bar loop.** For every tagged entry in the incoming `Snapshot<Sym>` (each `(symbol, freq, atom)` where `symbol` is `Some`), iterate `wallet.update(symbol, atom.candle)`, route each fill to `strategy.on_fill`, and append it (bar-tagged) to the report's blotter — one mark-to-market call per symbol that ticked this bar. Untagged entries (`Snapshot::of_atom` or `From<Atom>` / `From<Candle>` lifts) are skipped for wallet pricing but still visible to the strategy for indicator use. Then `strategy.update(snap)`, then `strategy.trade(wallet)` **iff `strategy.is_ready()`** (safe-by-default gate; trait default `true`), then push `wallet.equity().0` to the equity curve.

**Signature.** `run<Sym, S, W, I, A>(strategy, wallet, snapshots)` where `Sym: Clone + PartialEq`, `S: Strategy<Symbol = Sym, Input = Snapshot<Sym>>`, `W: Wallet<Sym>`, `A: Into<Snapshot<Sym>>`. Pass `Vec<Snapshot<Sym>>` for tagged input; `Vec<Atom>` / `Vec<Candle>` produce untagged size-1 snapshots (visible to strategy, wallet skips). Single-series callers wanting the wallet priced use `Snapshot::single(sym, atom)`.

**Report.** `RunReport<Sym> { equity_curve: Vec<Real>, fills: Vec<Fill<Sym>>, initial_equity: Real }` — one equity entry per input snapshot, `fills` records every booked `Order<Sym>` in a `Fill<Sym> { bar, order }`, `initial_equity` = `wallet.equity()` before the first bar (the seed `total_return` / `cagr` compound against; `= cash` for a fresh `PaperWallet::new(cash)`).

The CLI's `run()` / `evaluate()` (`src/cli/backtest.rs`) sits on this — wrapping each `Atom` in `Snapshot::single(spec.symbol.clone(), atom)` — with all CSV writing and console reporting in a post-run pass over `report.fills` / `report.equity_curve`. `Fill` and `RunReport` are re-exported at the crate root; `run` stays namespaced (`fugazi::backtest::run`) so the top-level `run` name isn't monopolized. A caller needing trade-streaming callbacks, per-bar hooks, or custom bar timing writes the loop directly — the primitive is deliberately opinion-free.

### Metrics = one function per metric (`src/metrics.rs`)

**There is no aggregate `compute`.** Every metric is its own `pub fn` in `fugazi::metrics`, so a caller picks whichever numbers matter and calls those; a caller that wants all of them calls all of them. Metrics that share an expensive intermediate consume public data types — three **intermediate builders** turn `RunReport` artefacts into what each metric family needs:
- **`per_bar_returns(equity: &[Real], initial_equity: Real) -> Vec<Real>`** — `(equity[i] - prev) / prev`, seeded from `initial_equity`; consumed by every return-moment / risk-adjusted metric.
- **`reconstruct_trades<Sym>(fills: &[Fill<Sym>]) -> Vec<Trade>`** — walks the blotter with one signed position and a volume-weighted entry; opposite-side fills close (or reverse); one closed leg = one `Trade { entry_bar, exit_bar, side, units, entry_price, exit_price, pnl, return_ratio }` (`bars_held()` = `exit_bar - entry_bar`). Consumed by every trade-level metric.
- **`drawdown_segments(equity: &[Real]) -> Vec<DrawdownSegment>`** — one peak → trough → recovery-or-end segment per drop; `DrawdownSegment { peak_bar, trough_bar, depth_ratio, duration_bars, underwater_bars }` — `underwater_bars` counts bars strictly below the peak so a caller sums them for `time_in_drawdown_ratio` without re-walking.

The metric catalogue itself is the standard set — return moments (`total_return`, `cagr`, `mean_return`/`median_return`/`stddev_return`/`best_return`/`worst_return`, `positive_bars_ratio`, `skewness`, `kurtosis`, `value_at_risk`/`conditional_value_at_risk` at any confidence, `tail_ratio`, `annualized_return`/`annualized_volatility`), risk-adjusted (`sharpe`, `sortino`, `calmar`, `omega`, `ulcer_index`, `ulcer_performance_index`), higher-moment / multiple-testing Sharpe corrections (`probabilistic_sharpe` — Bailey/LdP 2012 PSR against a caller-supplied benchmark Sharpe; `deflated_sharpe` — Bailey/LdP 2014 DSR against the expected max-SR under a normal null across `n_trials` grid candidates; each also has a `*_from_stats` variant that takes pre-aggregated Sharpe/skew/kurtosis so the `optimize` CSV writer can compute DSR per row from a `Metrics` document without re-scanning returns), drawdown (`max_drawdown`, `max_drawdown_duration`, `average_drawdown`, `average_drawdown_duration`, `drawdown_count`, `time_in_drawdown_ratio`, `recovery_factor`), and trade-level (`total_trades`, `winning_trades`/`losing_trades`/`flat_trades`/`long_trades`/`short_trades`, `max_consecutive_wins`/`max_consecutive_losses`, `win_rate`, `profit_factor`, `payoff_ratio`, `expectancy`, `kelly_fraction`, `average_win`/`average_loss`/`largest_win`/`largest_loss`, `average_trade_return`, `average_bars_held`/`min_bars_held`/`max_bars_held`, `exposure_ratio`). **Values are in natural units** (`0.15` = +15% total return, not 15.0) — the presentation layer scales to percent. Ratios whose denominator can vanish return `Option<Real>` and read `None` on the degenerate case (zero variance for Sharpe, no losing trade for a profit factor, non-positive endpoints for CAGR, …); metrics that are always well-defined return `Real` and read `0.0` on empty input. The PSR/DSR pair uses `statrs` for the standard-normal CDF (Φ) and its inverse (Φ⁻¹) — the one place a closed-form primitive wasn't practical without vendoring an Abramowitz-Stegun approximation.

**The library core stays lean** — no `Metrics` struct in the library at all (no serde on that layer). Optional deps come in behind three default-on features: **`sources`** (`reqwest`/`tokio`/`thiserror` for remote providers), **`runtime`** (pure, no-extra-deps type-erasure vocabulary in `fugazi::runtime` — `DynIndicator` / `DynValue` / `Adapter` / typed views), **`cli`** (implies both; adds `clap`/`clap_complete`/`serde_norway`/`csv`/`anyhow`/`indicatif`/`console`/`rayon`/`interim`). All pure Rust — no system libs — so the wheel path is uniform. The Python wheel builds with `default-features = false, features = ["sources"]`. The binary has `required-features = ["cli"]` so `cargo build` just works. **No plotting.** The CLI emits data files only (`trades.csv`, `returns.csv`, `metrics.yml`; under `-w LEN`: `metrics.csv` + `rolling.csv`); charts are downstream.

**CLI `Metrics` document** (`src/cli/metrics.rs`) carries the serde derives + `skip_serializing_if` + YAML field names (`sharpe`, `max_pct`, `annualized_mean_pct`, …). Populated by `metrics::from_report<Sym>(&RunReport<Sym>, bars_per_year, risk_free_rate) -> Metrics` — one library function per field, scaling to `_pct` at the assignment site. Everything downstream stays in the CLI:

- **`MetricKey`** — validated-once dotted-path handle; `MetricKey::from_name(name, sample)` + `.resolve(&Metrics) -> Result<Option<Real>>`. Reuse across rows to avoid re-walking the JSON tree.
- **`report_slice`** — sub-run over a bar range (equity slice, rebased fills, prior-bar initial equity); the shared measurement primitive.
- **`windowed_from_report`** / **`rolling_from_report`** — twin reductions (non-overlapping N-bar windows vs. rolling stride-1). Under `-w/--windowed LEN`, `run` emits both to `metrics.csv` / `rolling.csv` via `flatten` (fixed dotted-name column list held in lock-step with the document by a test); `metrics.yml` is still written as the whole-run summary. Console prints a second **windowed metrics** block after the whole-run one showing cross-window `mean ± std` on the headline stats.
- **`optimize -w`** uses only the non-overlapping variant: each `-m` metric becomes `_mean`/`_std` columns via `mean_std`; `--best-by` ranks by the windowed mean shifted by `-k/--risk-aversion` stddevs, direction-aware (`−k·std` for higher-is-better, `+k·std` for lower-is-better). Rolling is for plotting a continuous curve, not cross-window aggregation.
- **`selection.deflated_sharpe` column on `optimize`** — per-row DSR against the grid-wide null (`N` = number of trials, `Var[SR]` = sample variance of the grid's annualized Sharpes; computed once in `compute_dsr_context`). Whole-run mode uses each row's `metrics.risk_adjusted.sharpe`; windowed mode uses the arithmetic mean of window Sharpes (skew/kurtosis aggregated the same way, `n_returns` = sum of window bar counts). Emitted in both modes since windowing regularizes but doesn't eliminate selection bias. Omitted only when <2 rows have a defined summary Sharpe or the trial variance is zero.

`run` and `optimize` measure over the whole run; the metrics stack has no crop of its own. The strategy readiness gate holds the first trade until every consulted source is past its `stable_period()`, so `total_return` / `cagr` / `sharpe` reflect that shape naturally. (`get`'s `--keep-unstable` flag is the sibling pattern one level up — see [Safe defaults, opt-in overrides](#safe-defaults-opt-in-overrides).) `Trade` and `DrawdownSegment` are re-exported at the crate root; metric functions stay namespaced (`fugazi::metrics::sharpe(&returns, rf, bpy)`, etc.).

### Generic transform ops (`src/indicators/ops.rs`)

Source-wrapping carriers, each driven by an operator type so a new operator is a trait impl, not a new type:
- `Combine<L, R, Op>` (binary, `BinaryOp`): **one carrier for all binary ops**, generic over the op's input/output via associated `Lhs`/`Rhs`/`Output` and holding the op **by value**. Serves arithmetic `Add`/`Sub`/`Mul`/`Div` (`Real,Real→Real`; `Div` → `None` on /0), the comparisons in `indicators/compare.rs` (`Real,Real→bool`, the op carrying its epsilon) and the boolean logic in `indicators/logic.rs` (`bool,bool→bool`). `Combine::new` needs `Op: Default`; comparison ops also get `Combine::with_epsilon`.
- `Lookback<I, Op>` (unary, relates a source to its value `period` steps ago, `LookbackOp`, zero-sized markers): `Lag` (past value), `Diff` (`x[t]-x[t-n]`), `Ratio` (`x[t]/x[t-n]`), `Roc`.
- `Extreme<S, Op>` (rolling extremum, `ExtremeOp` = `MaxOp`/`MinOp`): `RollingMax`/`RollingMin`.

**`IfElse<Cond, T, F>`** (`src/indicators/if_else.rs`) is the three-source ternary — the idiomatic branching primitive. `Cond: Indicator<Output=bool>` picks between two real sources: `Some(true)` → `if_true`'s value, `Some(false)` → `if_false`'s, `None` on the cond → `None` propagates. All three sources are advanced every bar (never short-circuited on the non-selected branch, matching `Combine`'s convention), so a branch that doesn't fire keeps its warm-up progressing. `warm_up_period()` and `stable_period()` report the max across the three (safe worst case for downstream stability gates), but **the actual first `Some` can arrive earlier**: as soon as the condition and the *selected* branch are both settled, even if the unselected one still isn't. This is intentional — `warm_up_period()` is an *upper bound* on first-`Some`, not the exact position — so `IfElse` is deliberately excluded from the `tests/warm_up.rs` exact-warm-up battery (which asserts "first `Some` at exactly sample N"). Fluent builder: `.if_else(t, f)` on `BoolIndicatorExt`, reads left-to-right: `adx.gt(25.0).if_else(roc, Value::new(0.0))` is the ADX-gated momentum score in one expression. YAML: `!if_else { cond, if_true, if_false }` on `ExprSpec` (`cond` is a `SignalSpec`, both branches are `ExprSpec`). Python: `ta.if_else(cond, if_true, if_false)`.

### Shared cores (`pub(crate)`)

Bare `Real -> Real` math with **no source and no `Indicator` impl**, so both source-wrapping indicators and indicators smoothing values they compute *internally* share one implementation:
- `smoothing.rs`: `EmaState` (EMA recurrence) and `WilderState` (Wilder/RMA, mean-seed). `Ema`/`Macd` use `EmaState`; `Rma` uses `WilderState`; `Rsi` uses two (gain/loss); `Atr` = `TrueRange` + `WilderState`; `Adx` uses four.
- `stats.rs`: `WindowStats` (windowed sum + sum-of-squares → `mean`/`variance`/`stddev`) backs `Sma`/`StdDev`/`Bollinger`; `WindowExtreme<Op>` (monotonic-deque rolling extremum) backs `Extreme`/`RollingMax`/`RollingMin` and `Stochastic`.

<a id="the-two-provider-traits"></a>### Remote providers = two traits, by shape (`src/sources/`)

Behind the `sources` feature. Both traits are `Send + Sync`, use edition-2024 RPITIT (`impl Future`, not `async fn`) so callers can name the `Send` bound, take **objects/enums, not strings** (the CLI and Python parse user text before calling in), and share one `SourceError` and one `Interval` cadence enum.

- **`CandleSource`** — `atoms(symbol, interval, since, until) -> Vec<Atom>`, ascending by `Atom::time`. Impls: `Binance` (`binance_schema()`: `quote_volume` / `n_trades` / `taker_buy_base_volume` / `taker_buy_quote_volume`), `Yahoo` (`yahoo_schema()`: `adj_close`). Every atom in one fetch shares one `Arc<Schema>` — pick it off a returned slice with `schema_of`.
- **`OverlaySource`** — `overlays(symbol, interval, since, until) -> Vec<OverlayRow>`, plus a `schema()` that is stable before any fetch. For providers of **per-bar side-channel data with no OHLCV**: market cap, supply, open interest, funding, sentiment. Two impls: `CoinGecko` (`coingecko_schema()`: `price` / `market_cap` / `total_volume` / `circulating_supply`, the last derived as `market_cap / price`) and `CoinMarketCap` (`coinmarketcap_schema()`: `price` / `volume_24h` / `market_cap` / `circulating_supply` / `total_supply`; **paid tier only** — historical quotes gate behind a paid CMC plan, auth via the `X-CMC_PRO_API_KEY` header from `CMC_PRO_API_KEY`; unlike CoinGecko it takes an explicit `interval` so it fetches the requested cadence directly instead of bucketing an over-sampled response, then still floors timestamps to the bar-open boundary for the join).

**Why two traits and not a capability flag.** `Atom::candle` is not `Option`, and `Wallet::update` marks a symbol's price from the bar it is fed — so an overlay provider forced through `CandleSource` would have to synthesise a candle, and that fake OHLCV would flow into `Current::close()` and into mark-to-market. `OverlayRow { time, overlays }` has no candle *field*, which makes the mistake unrepresentable rather than merely discouraged. A provider that genuinely has both is free to impl both traits. `OverlayRow`'s equality/ordering is **by `time` alone**, exactly like `Atom`'s (the payload is a bag of `f64`s that may hold `NaN`, so comparing it would be neither reflexive nor useful).

**How overlay data reaches a strategy.** An `OverlaySource` yields columns, not a tradeable series — so you fetch it to its own CSV and let the `--series` dataframe join it onto a price series by `(symbol, time)` (`DataFrame::insert`, `src/cli/data.rs`). The joined columns land in the run's `Schema` and are read with `!get { key: market_cap }`:

```text
fugazi get binance:BTCUSDT[1d]                       -o prices.csv
fugazi get cg:BTCUSDT=bitcoin[1d]                    -o caps.csv
fugazi run @strategy.yml -s @prices.csv -s @caps.csv -o out/
```

The `OUT=QUERY` remap is what makes the join key line up (see [`get.rs`](#cli-internals-srccli)). Cross-sectional `BasketStrategy` scores are the natural consumer — market-cap rank, supply-adjusted momentum, and dominance filters are not derivable from OHLCV at all.

**CoinGecko specifics worth knowing before reaching for it.** `market_chart/range` has no interval parameter — CoinGecko picks the sampling granularity from the *window length* (~5-minutely ≤1d, hourly ≤90d, daily beyond). So the client rejects sub-hourly `Interval`s outright, paginates hourly requests in 80-day windows to stay inside that tier, and buckets whatever samples arrive onto the requested cadence's bar-open boundaries keeping the **first** sample per bucket (the value as of the open — never a later reading a strategy couldn't have seen). Weekly buckets floor to Monday and monthly to the 1st via the calendar, not epoch modulo (epoch day 0 was a Thursday, which would silently fail to join against a Monday-opening weekly candle). A descriptive `User-Agent` is **mandatory** (they 403 without one), and the public tier serves only the **last 365 days** — a wider `--since` surfaces as an `Http` error rather than being silently truncated. `COINGECKO_API_KEY` is picked up as a demo key.

## Safe defaults, opt-in overrides

Numbers this crate produces during a source's warm-up or IIR settling tail are *unsettled*: they exist, but they depend on the seed, on where the window happens to start, or on both. Every knob that could paper over an unsettled bar therefore biases toward **waiting**, with a single-name **flag / wrapper / period** for the caller who's considered the tradeoff and wants to trade through it. Current instances of the pattern:

- **Strategy readiness.** `Strategy::is_ready()` (default `true`) gates `trade()` in `fugazi::backtest::run`; `SingleAssetStrategy` overrides it to `bars_seen >= max(stable_period across every wired signal, attached level, and the sizing indicator)`. Opt-out: `Unstable<S>` (Rust `.unstable()`, YAML `!unstable { source | signal }`, Python `.unstable()` / `ta.unstable(x)`) is a transparent passthrough that reports `unstable_period() = 0` for the wrapped subtree.
- **Position sizing.** `SingleAssetStrategy::position_sizing(indicator)` and `PairsStrategy::position_sizing(indicator)` (each defaulting to `Value::new(1.0)`) scale the `value_frac` magnitude on every entry/reversal — Kelly, vol-targeting, fixed-fractional. On the pair, both legs are scaled together (`value_frac(0.5 * m)` per leg). A `None` reading from the sizing indicator *skips the whole `trade()` call* for that bar (safe default; the caller opts out by composing a well-defined fallback into the spec, e.g. wrapping the sizing subtree in a max-against-`Value::new(1.0)`). YAML surface: top-level `sizing:` on both `SingleStrategySpec` and `PairsStrategySpec`. The crate ships five ready-made sizing recipes in `fugazi::indicators::sizing`, split by what state they read: **price-based** (no strategy anchor needed) — `vol_target(target, window, bars_per_year)` (inverse realized-vol / vol targeting) and `atr_risk(risk_frac, period, atr_multiple)` (fixed per-trade risk sized by ATR); **book-anchored** (read the strategy's `Book` — see [Book anchor](#book-anchor)) — `drawdown_throttle(&book, max_drawdown)` (linear de-lever as drawdown deepens, clamped `[0, 1]`), `equity_vol_target(&book, target, window, bpy)` (vol targeting on the strategy's own equity return stream), and `fractional_kelly(&book, kelly_fraction, window)` (continuous-return Kelly over the last N closed trades, clamped `>= 0`). YAML tags for all five: `!vol_target`, `!atr_risk`, `!drawdown_throttle`, `!equity_vol_target`, `!fractional_kelly`. All three book-anchored recipes work on both `SingleAssetStrategy` and `PairsStrategy` — on a pair the book tracks the aggregate two-leg equity curve, per-trade P&L sums both legs' contributions.
- **`fugazi get` overlays.** The CLI trims each overlay column's pre-`stable_period()` cells before writing the CSV. Opt-out: `--keep-unstable`, which emits every sample from bar 1.
- **`-w/--windowed` duration form.** `-w 1d`/`-w 1w`/… resolves against the trading calendar (see `WindowSpec::resolve`), and demands both an explicit `AssetClass` (`--stocks`/`--forex`/`--crypto`) and a resolvable bar cadence — a silent Stocks default would change a window's bar count in a way that's hard to notice on 24/7 data, and a missing cadence has no defensible fallback. Opt-out: the plain bar-count form `-w N`, which needs neither.
- **Explicit periods.** Every windowed constructor takes an explicit `period` (asserted `> 0`); risk-adjusted metrics take an explicit risk-free rate and bars-per-year. No "sensible defaults" that would hide the choice.

When adding a new knob that touches unsettled data: pick the value that's safest when the user forgets to think about it, and provide *one* mechanism to opt out. Don't add a wrapper whose default flips the polarity — the point of the pattern is that reading a spec bottom-to-top tells you what a distracted operator gets, and the exceptions are named right there in the tree.

## Conventions and gotchas

- **Composition is construction.** A new "X of Y" indicator takes its source `S` in `new` (or `of`, for source-generic leaves whose zero-arg `new()` uses the default `Identity<Atom>` root — see the [Leaf sources](#indicators--the-numeric-sources-srcindicatorrs-srcindicators) bullet) — with the right output constraint (`S: Indicator<Output = Real>` for numeric wrappers like `Ema`/`Sma`, `S: Indicator<Output = Candle>` for bar indicators, `S: Indicator<Output = Atom>` for the atom-input leaves); don't add pipe combinators.
- **Use the cores, not each other's public types.** Internal smoothing of computed scalars uses `EmaState`/`WilderState` (Real recurrence). The public `Rma<S>`/`Ema<S>` wrap a *source* and can't smooth values you computed inline.
- **Adding an operator** (arithmetic/comparison/boolean/lookback): add an `*Op` type implementing the relevant `*Op` trait plus a type alias — never a macro. Arithmetic/boolean/lookback ops are zero-sized `Default` markers; comparison ops are value structs carrying `epsilon`. They live by their carrier — `indicators/{compare,logic}.rs` for the `Combine`-based ones, `indicators/ops.rs` for `Lookback`/`Extreme`.
- `Combine` (arithmetic, comparisons, and `And`/`Or`/`Xor`) feeds the *same* input to both sides, so it requires `Input: Clone`. Use `lhs`/`rhs` naming for binary operands.
- `Combine` holds its op **by value** (so a comparison can carry epsilon); `Lookback`/`Extreme` hold a zero-sized op as `PhantomData<fn() -> Op>`. Input-ignoring / field-marker leaves (`Value<I>`, `Const<I>`, the `F` in `Field<F, S>` / `Calendar<F, S>`) use `PhantomData<fn(I)>` / `fn() -> F` to satisfy the constraint rules (avoids E0207); `Identity<I>` similarly uses `PhantomData<fn(I) -> I>` for the pinned input=output relationship.
- `Change` is a **bidirectional** toggle detector (fires on any transition); directional events come from pairing it with a comparison (see `crosses_above`).
- Constructors `assert!(period > 0, ...)`; document warm-up length in the type's doc comment, and implement `warm_up_period()` to match it exactly (plus `unstable_period()` when the indicator smooths recursively).
- A comparison/edge is **`None` until** every source it depends on is warmed up (it reads `false` via `.is_true()`); a boolean op (`And`/`Or`/…) is likewise `None` until both sources are ready, so an edge that would coincide with warm-up is not detected (no spurious first-bar trade).

## CLI internals (`src/cli/`)

The CLI is one binary (`fugazi`) with several subcommands; layout by concern:

- **`main.rs`** — clap definitions, subcommand dispatch. Owns the `mod` declarations and the top-level `Cli` type.
- **`run.rs`, `optimize.rs`, `backtest.rs`** — the two user-facing subcommand drivers (`run`, `optimize`) sit on the pure `backtest` module (`run_iteration`, `evaluate`, `evaluate_windowed`). `backtest.rs` deliberately owns no IO; `run.rs` / `optimize.rs` do the CSV writing, console output, and file paths.
- **`get.rs`** — the `fugazi get` subcommand (fetch bars or overlay columns from remote providers). Grammar: `<provider>:[OUT=]<symbol>[[OFREQ=]<freq>,...]` (`parse_spec` / `parse_symbol`). Both remaps read **left = emitted, right = fetched**, and both are optional (absent ⇒ the two are the same):
  - `OUT=` decouples the `symbol` cell from the identifier sent to the provider (`cg:BTCUSDT=bitcoin[1d]` fetches the coin id `bitcoin`, writes `BTCUSDT`). This is what makes a `--series` join line up, since that join is an exact string match on `(symbol, time)`.
  - `OFREQ=` decouples the `freq` cell from the fetched cadence (`[1d=24h]`, `[FOO=1d]`). It **relabels, it does not resample** — every fetched row is emitted, only the `freq` cell is rewritten, so mapping across *different* durations (`[4h=1h]`) produces rows no price bar can join and `run` rejects them. The emitted label is an **opaque string** (only the fetched side is parsed into an `Interval`), which is safe because `data.rs` treats `freq` as a reserved passthrough column and never parses it. `csv_source.rs` *does* parse it, so a file labelled `FOO` can't be re-read through `get csv:PATH`.
  - Two pipelines, picked by `resolve_mode` and **never mixed in one invocation**: `run_candles` (OHLCV + `-x` overlays → `write_candles_csv`) and `run_overlay_columns` (an [`OverlaySource`](#the-two-provider-traits) → `write_overlays_csv`, which emits `symbol,freq,time,<columns>` and **no OHLCV block**). Mixing is a hard error because the `--series` join lets the later file win each column, so a synthetic zero-OHLCV block would silently overwrite real prices. `-x` is likewise rejected in overlay mode — a computed overlay is an indicator chain over `Atom`s and there are no candles to build one from.
  - `get` carries its own **`--params`** flag (`params::ParamSpec`, same grammar as `run --params`): it resolves `!param` placeholders **inside the `-x/--overlay` expressions** (via `params::table` → `overlay::parse_specs(specs, &param_table)`), so `--params FAST=20 -x 'ma=!sma { period: !param FAST }'` parameterizes an overlay the way it parameterizes a strategy doc. It only touches overlay bodies — a fetch with no `-x` has nothing to substitute — and is ignored by overlay-only providers that define no computable columns.
- **`spec/`** (directory) — YAML-deserializable mirror of the fugazi composition API, one file per layer. Three composable expression types and three top-level strategy documents:
  - `spec/expr.rs` — `ExprSpec` (value-producing expression enum; nominally `Real`, polymorphic over `DynType` for `!current`/`!pick`/`!time`/`!get`/`!if_else`/`!value`). Includes the `default_source`/`default_high`/`default_low`/`default_bar_source` helpers (all `pub(super)`) and **`ValueLit`** — the `!value` payload, a number (→ `Value`, `Real` output) or a string (→ `ValueStr`, `Str` output; `!value bull`). Quoting picks the type when they'd collide (`!value 70` scalar vs `!value "70"` string). `ValueLit` deserializes through the `serde_norway::Value` bridge, not `#[serde(untagged)]` — serde's untagged content buffering can't see YAML tags, and the bridge gives a real "!value takes a number or a string" error.
  - `spec/signal.rs` — `SignalSpec` (boolean condition enum + `eps` tolerance helper) + **`StrOperand`** — the `rhs` of `!str_eq` / `!str_ne`: a bare string literal (`rhs: bull`) *or* any `Str`-output `ExprSpec` (`rhs: !value bull`, `rhs: !get { key: prev_regime }` — column-vs-column). Same `serde_norway::Value` bridge, and for the same reason: `ExprSpec` carries its own tag normalisation that untagged buffering would bypass. A literal builds the same `ValueStr` the `!value` form does.
  - `spec/template.rs` — **`SpecTemplate<T>`**: the third composable YAML type alongside `ExprSpec` / `SignalSpec`. Captures a raw `serde_json::Value` at deserialize time; `.build(&args)` runs `!arg` substitution then typed-parses. **Untagged in YAML** — the template-ness is a schema fact of the containing struct's field type, so a user writes a normal `T`-shaped tree with `!arg SYM` leaves inside. Two-pass substitution: `!param` runs *once at load* from `--params` (via `crate::params::substitute`, before construction), `!arg` runs *each `.build()`* from a driver context (via `crate::args::substitute`). Keyed on distinct singleton-object keys (`param` vs. `arg`) so they never collide. Currently consumed by `BasketStrategySpec::{score, sizing}` for per-symbol substitution.
  - `spec/strategy.rs` — `SideSpec`, **`SingleStrategySpec`** (single-asset), `DynSingleStrategy` (the built strategy wrapper; `from_single` wraps a pre-built `SingleAssetStrategy<String>` for the preset path).
  - `spec/preset.rs` — **`StrategyPreset`** (externally-tagged sugar for the `fugazi::strategies` catalogue: `!buy_and_hold` / `!ma_crossover` / `!rsi_reversal` / `!donchian_breakout` / `!keltner_breakout`; each variant delegates to the Rust recipe — single source of truth, no re-encoding) and **`StrategyRef`** (`Spec(SingleStrategySpec) | Preset(StrategyPreset)`, a `serde_norway::Value`-bridge enum that routes a preset tag/single-key map to `Preset` and anything else to `Spec`). `StrategyRef` is what the top-level Single `run`/`check` path and the trailing tags' `strategy:` field both load; `run::run` / `backtest::run_iteration` thread `&StrategyRef` (its build closure covers both arms). `optimize` stays `SingleStrategySpec`-only (presets carry no `!param` axes).
  - `spec/trailing.rs` — the CLI builder for the trailing risk tags (`!sharpe` / `!sortino` / `!volatility` / `!max_drawdown` / `!calmar`). Wraps the non-`Clone` `Sharpe<DynSingleStrategy>` etc. in a `RebuildIndicator` that rebuilds the strategy from its `StrategyRef` on clone (the erasure layer requires `Clone`; `SingleAssetStrategy` shares its `Position`/`Book` via `Rc`, so a fresh rebuild is the only independent clone).
  - `spec/pairs.rs` — `PairsStrategySpec` (two-symbol pairs), `DynPairsStrategy`.
  - `spec/basket.rs` — **`BasketStrategySpec`** (N-symbol cross-sectional basket) + **`SelectionRuleSpec`** (externally-tagged: `!top_bottom { longs, shorts }` / `!threshold { long_min, short_max }` / `!quantile { long_q, short_q }`; CLI-side mirror of the library enum, dispatches to the three free functions `top_bottom` / `threshold` / `quantile` at build time). Fields: `selection`, `score: SpecTemplate<ExprSpec>`, `sizing: SpecTemplate<ExprSpec>`. `.build(initial_equity, schema)` clones the two templates into per-symbol factories that resolve `!arg SYM` on every new symbol seen in the snapshot. The `!entry` / `!peak` / `!trough` position accessors read a dummy `Position` inside score/sizing subtrees (always `None`) — meaningful only inside per-side protective levels, which BasketStrategy doesn't wire yet. The shared `Book` anchor IS wired, so `!drawdown_throttle` / `!equity_vol_target` / `!fractional_kelly` work per-symbol against the basket's aggregate equity curve. **`!equal_weight <N>`** is the one-line sizing sugar for a 100%-gross balanced basket (`sizing: !equal_weight 6` on a 3-long/3-short → 1/6 per leg). **Wired into `run.rs`** (`basket:@basket.yml` prefix routes through `run::run_basket`, mirroring `run_pairs`); `optimize` still bails on baskets (grid-sweep on templates is a follow-up).
  - `spec/mod.rs` — module doc, `pub use` re-exports, the shared **`load_value(text, params, base)`** pass (`parse → !import → !param → typed parse`, the one place that order is written down; all three `*StrategySpec::from_text_with_params_in` sit on it), and the tests for the whole spec surface.
- **`costs/`** (directory) — `--costs` parsing and resolution:
  - `costs/spec.rs` — CLI-arg parsing into `CostSpec` (a term list); `CostTerm` variants + `split_top_commas` / `parse_term`.
  - `costs/config.rs` — everything downstream: normalization, `CostConfig`, `LegConfig<T>`, `ScopedEntry<T>`, the typed `CommissionSpec`/`SpreadSpec`/`SlippageSpec` model enums, and `impl CostConfig::resolve`. The model enums are **externally tagged** like every other YAML-visible enum in the project — a model is `!percentage { rate: 0.001 }`, never `kind: percentage`. Consequently a dotted `--costs` setter is a *literal* address into the spec tree, variant level included (`commission.percentage.rate=0.00075` nudges a loaded preset; `commission=!percentage { … }` replaces the model). Naming the wrong variant is a hard error (`check_variant`), not a silent half-application. `MODEL_VARIANTS` + `is_model` are what let the untyped passes (leg hoisting, deep merge) recognize a model node.
  - `costs/mod.rs` — module doc, `pub use` re-exports, tests.
- **`dyn_indicator.rs`** — one-line facade re-exporting **[`fugazi::runtime`](../src/runtime.rs)**, the runtime-typed indicator handle the YAML builder produces. The vocabulary itself — `DynIndicator` trait + `DynValue` payload enum (`Real | Bool | Atom | Candle | Str | Time | Snapshot<String>`) + `DynType` tag + `Adapter` blanket + `AsReal`/`AsBool`/`AsCandle`/`AsAtom`/`AsStr` typed views + `chain`/`unstable_wrap` composers — now lives in the library behind the `runtime` Cargo feature (default-on; implied by `cli`) so YAML/JSON-driven builders and downstream consumers can share it. Callsites in `src/cli/spec/*` still say `crate::dyn_indicator::*` unchanged. **Any new YAML-visible indicator plugs in via `dyn_indicator::wrap(...)` → `fugazi::runtime::wrap(...)` here.**
- **`csv_source.rs`** — local CSV candle source for `fugazi get csv:PATH`; sibling to `sources/binance.rs` and `sources/yahoo.rs` conceptually, but CLI-side because it owns the CSV parsing.
- **`data.rs`** — the `--series` data frame builder (multiple `@file.csv` + inline literal specs, full-joined on `symbol` + `time`). Uses `crate::csv_source::detect_delimiter` (the shared CSV delimiter probe).
- **`overlay.rs`** — `--overlay` parsing for `fugazi get`. Its scope type (`OverlayScope`) uses `sources::Interval` (not `Frequency`) because a get scope binds to a remote-fetch interval, so it stays its own type — but the bracket-splitting delegates to `calendar::parse_scope_parts` (see below).
- **`calendar.rs`** — asset-class + bar-frequency vocabulary. Owns `Frequency`, `AssetClass` (`trading_days_per_year` / `trading_hours_per_day` / `trading_seconds_per_bar`), `Scope { symbol, freq }`, `ScopedFrequency`, and the shared **`parse_scope`** + **`parse_scope_parts`** helpers. Also owns **`WindowSpec`** (`-w/--windowed` grammar: `Bars(NonZeroUsize) | Duration(Frequency)` — plain count or `m/h/d/w/M` suffix, resolved via `WindowSpec::resolve(bar_freq, class)` = `win.trading_seconds / bar_freq.trading_seconds`; duration form errors on missing `class` or `bar_freq` — see [Safe defaults](#safe-defaults-opt-in-overrides)), **`parse_time_to_millis`** (RFC3339 / `YYYY-MM-DD [HH:MM:SS]` / epoch s or ms → UTC ms; consumed by `data.rs` at load to populate `Atom::time`), and **`detect_frequency_from_atoms`** (reads `Atom::time` directly, no re-parse).
- **`metrics.rs`** — the CLI's aggregate `Metrics` document + `MetricKey` handle + `resolve_metric` wrapper. Details in the metrics section above.
- **`input.rs`** — the `@file`-or-inline `Source` type used by every text-taking flag (`--params`, `--overlay`, `--costs`, the strategy positional). Also owns **`base_dir()`** — the directory a document's relative `!import` paths resolve against (the file's own directory for `@path`, the working directory for inline text).
- **`glob.rs`** — shell-style glob matching (`Pattern: FromStr` + `.matches(text)` + `.is_anchored()`), used by `fugazi list tickers <provider> [PATTERN]` to filter a provider's vocabulary (`'b*'` starts-with, `'*b*'` contains, `?` one char, `[a-z]` / `[!abc]` classes, `\*` literal). **Case-insensitive and whole-string** — the provider owns the casing (`BTCUSDT` vs `bitcoin`) and a glob means whole-string everywhere else a shell user meets one, so "contains" is spelled `*b*`. Matching is a linear two-pointer scan with one backtrack point per `*`, so `foo*bar*baz` and `*a*a*a*a*b` are both fine. Hand-rolled on purpose: translating to a regex would add `regex` + `regex-automata` + `aho-corasick` at runtime (the `regex-syntax` in `cargo tree` is build-time only, under `interim`'s proc-macro) and would still need the class parsing, leaving only the ~25-line scan to delete. Filtering lives here, not in the source traits — no provider endpoint offers a server-side filter, so a `pattern` parameter there would be a lie every impl re-implements.
- **`imports.rs`** — the `!import` pass: `resolve(value, base)` walks the untyped tree and replaces every `!import <path>` node with the **whole value of the document it names**, so any subtree (a signal, a sizing recipe, a whole `long:` side) can live in its own file. Third member of the singleton-object-walker family alongside `params`/`args`, and keyed on its own sentinel (`import`). Two rules worth knowing: paths resolve **relative to the importing document**, not the process CWD (a nested import inside an imported file therefore re-bases on *that* file's directory), and imports run **before `!param`**, so an imported fragment sees the same `--params` table as its importer. Cycles are a hard error naming the chain (canonical-path stack), not a stack overflow. `optimize` resolves imports **once** into its base value, so a shared fragment costs one read no matter how big the grid.
- **`params.rs`, `args.rs`, `convert.rs`, `list.rs`, `completions.rs`, `pool.rs`, `style.rs`** — auxiliary: `!param` substitution (load-time, from `--params`), `!arg` substitution (build-time, from a driver context — see `SpecTemplate<T>`), YAML→JSON normalization, `fugazi list`, shell completions, rayon pool builder, terminal styling. `params::substitute` and `args::substitute` share the same structural shape (singleton-object walker, bare-string or `{ key, default }` body) and differ only in the sentinel key. None are entry points; touch on demand.

## Python bindings (`python/src/lib.rs`)

The Python surface is a **type-erased mirror** of the Rust library, packaged as one pyo3 cdylib (`fugazi-python` crate → `fugazi` importable module). The core library is monomorphised over source generics (`Ema<S>` etc.); Python can't carry those across the FFI boundary, so the whole file is erase-then-dispatch. The erasure vocabulary is **`fugazi::runtime`** — `DynIndicator` trait exchanging `DynValue` payloads, plus the `DynIndicatorSync` subtrait adding pyo3-required `Send + Sync` bounds and an autotrait-preserving deep clone (via `runtime::wrap_sync`). Every output-typed carrier is a generic **`TypedSource<In, Out>`** newtype over `Box<dyn DynIndicatorSync>`, aliased on the output: `Source<I> = TypedSource<I, Real>`, `StrSource<I> = TypedSource<I, Arc<str>>`, `AtomBox<I> = TypedSource<I, Atom>`. `SignalBox<I>` is the one specialisation — a newtype around `TypedSource<I, bool>` that flattens warm-up `None` to `Some(false)`, so `.not_()` of a warming-up signal reads `true` (Python's promise: a signal always has a definite `bool`). Multi-output erasure stays local as `DynMulti<I>` / `MultiBox<I>` — its `Vec<Real>` + `&'static [&'static str]` shape doesn't fit `DynValue`, and unifying would need a library-side `Multi` variant for negligible savings. `AnySource` / `AnySignal` / `AnyMulti` enums record the input domain — candle-rooted (`Input = Atom`), value-rooted (`Input = Real`), snapshot-rooted (`Input = Snapshot<String>`) — and the `map_source!` / `combine_sources!` / `source_to_signal!` / `sources_to_signal!` / `map_signal!` / `combine_signals!` / `map_multi!` / `combine_multi!` macros dispatch on those tags. **Rule: a Python constructor mirroring a Rust one uses those macros; it never names a concrete `Ema<Sma<Current, …>, …>` type.**

### Parity discipline: keep Python in sync when the library grows

The Python bindings ship the same primitives as the Rust core, with three deliberate omissions (see below). **When a Rust API is added, extended, or renamed, mirror it in `python/src/lib.rs` in the same PR** — Python drift is otherwise silent (the Rust build still passes) and only surfaces when a user reports missing bindings.

The mirror rules by layer:

- **New indicator / signal / operator on the pure layer** → add a `#[pyfunction]` in `python/src/lib.rs`, register it in the `#[pymodule] fn fugazi` reg list, and add a smoke test in `python/tests/test_fugazi.py`. Single-output real-source constructors use `src_period!`; bar-only constructors use `bar_period!`/`bar_noarg!`; multi-output ones use `bar_period_multi!` or a hand-written `map_multi!`/`combine_multi!` body. If it's a new fluent method on `IndicatorExt` / `BoolIndicatorExt`, add a `#[pymethods]` entry on `PyIndicator` / `PySignal` alongside the existing ones (and update the `__add__`/`__and__` etc. dunders if it composes with operators).
- **New metric function in `src/metrics.rs`** → add a `#[pyfunction]` in the metrics section, add its name to the `register_metrics_module` reg list. Signature is mechanical: `Vec<Real>` for slice inputs, `Vec<PyTrade>` / `Vec<PyDrawdownSegment>` / `Vec<PyFill>` for the intermediate types (converted to the native `Vec<Trade>` etc. inside the body). Ratios returning `Option<Real>` stay `Option<Real>` on the Python side (materialises as `None`); metrics returning `Real` stay `f64`; bar counts stay `usize`.
- **New field on `Trade` / `DrawdownSegment` / `Order` / etc.** → add a `#[getter]` on the corresponding `PyTrade` / `PyDrawdownSegment` / `PyOrder` and update its `__repr__`.
- **New remote provider under `src/sources/`** → add a `Py*` client class, register it, and add a branch in the `fetch(provider=…)` dispatcher. Which shape depends on the trait: a **`CandleSource`** mirrors `PyBinance` / `PyYahoo` (one `.candles(...)` → OHLCV frame via `build_candles_frame`; `fetch` dispatches to it). An **`OverlaySource`** mirrors `PyCoinGecko` / `PyCoinMarketCap` (one `.overlays(...)` → `build_overlays_frame`, a `time` column plus the provider's schema keys and **no OHLCV**), and its `fetch` branch **redirects with an error rather than dispatching** — `fetch` is documented to return candle frames, so silently handing back a candle-less one from it would be a trap.
- **New leaf source / component accessor** → add a `#[pyfunction]` using `leaf!` or a `candle_source(...)` body, register it.
- **Changes to `Candle` / `Atom` / `OverlayInfo` / `Schema` / `SchemaBuilder`** → update `PyCandle` / `PyAtom` / `PyOverlayInfo` / `PySchema` / `PySchemaBuilder` field-for-field.

**Partially bound — the single-asset strategy builder + `run`.** `PyStrategy` (`Strategy` in Python; the "Strategy builder + run" section of `python/src/lib.rs`) mirrors `SingleAssetStrategy`'s declarative builder — `Strategy(sym).long_on(enter, exit).short_on(down, up).position_sizing(src).run(wallet, candles)` — driving `fugazi::backtest::run` over a `PyWallet` and returning a `PyRunReport` (`RunReport`, whose `.fills` are `PyFill`s the metrics functions consume). Python signals are candle-rooted; the strategy layer is snapshot-rooted, so `AtomLift` (sole-atom projection) bridges them. **Still not bound within this** (do not add without being asked): position-anchored protective levels (`Position` uses `Rc<RefCell>`, which isn't `Send + Sync`, so `entry()`/`peak()`/`trough()` can't be type-erased for pyo3 — a would-be `Arc<Mutex>` refactor), `PairsStrategy`/`BasketStrategy`, and the `src/strategies/` recipe catalogue (available in YAML as presets; a Python user composes signals imperatively). When extending `PyStrategy`, update `python/tests/test_fugazi.py` + the `python/README.md` "declarative `Strategy` builder" block.

**Intentionally not bound** (do not mirror without being asked):

- The lower-level `Strategy` *trait* (as a Python-subclassable interface) and the `src/strategies/` catalogue recipes as Python constructors. A Python user either uses the `PyStrategy` builder above or drives a `PaperWallet` imperatively.
- `run_iteration` / `evaluate*` (the CLI's windowed/optimize measurement pipeline). `backtest::run` + `RunReport` are bound (via `PyStrategy.run`); the CLI-only measurement stack is not.
- The **trailing strategy-risk indicators** `Sharpe` / `Sortino` / `Volatility` / `MaxDrawdown` / `Calmar` (`src/indicators/trailing.rs`; YAML `!sharpe`/`!sortino`/`!volatility`/`!max_drawdown`/`!calmar`). Each *owns a `Strategy`* and — because a built strategy shares `Rc`-anchored state and isn't cloneable/`Send`-friendly for the erasure layer — stays CLI/Rust-only by design.
- The CLI (`src/cli/`). It's a separate binary; no Python analogue is intended.
- The `Wallet` trait itself. Python only exposes `PaperWallet` — a live wallet would live in a downstream Rust crate implementing `Wallet` and be re-bound separately.
- Rust-internal types that only make sense inside the strategy layer (`Position`, `PositionField`, `Ack`, `OrderId`, `Reference`, `Units`). Python takes `float`s for prices and unit counts and returns `PyOrder` (with `bar` tagging done via `PyFill` when the user drives a metrics workflow).

Layout inside `python/src/lib.rs` (single file, ~3400 lines — grep by section header):

- **Type-erasing carriers** — `TypedSource<In, Out>` newtype over `Box<dyn runtime::DynIndicatorSync>`; aliases `Source<I>` / `StrSource<I>` / `AtomBox<I>`; `SignalBox<I>` (over `TypedSource<I, bool>`, warm-up flattening); local `DynMulti<I>` / `MultiBox<I>` (multi-output, doesn't fit the shared vocabulary); `MultiOutput` shim; `ResampleThen`.
- **Input domain** — `AnySource` / `AnySignal` / `AnyMulti` enums, `pair()` resolver, `domain_mismatch()`, and the `map_*` / `combine_*` macros. **All new binding bodies use these macros.**
- **Python classes** — `PyCandle`, `PySchema`, `PySchemaBuilder`, `PyOverlayInfo`, `PyAtom`, `PyIndicator`, `PySignal`, `PyMulti`, `PyWallet`, `PyOrder`, `PySize`.
- **Strategy layer** — `PyWallet` with `set` / `set_position` / `close` / `set_stop` / `set_take_profit` / `cancel_protective` / `update` (returns `Vec<PyOrder>` — a bar-loop that later hands orders to metrics wraps them in `PyFill(bar, order)`). Plus the declarative **`PyStrategy`** (`Strategy`) builder + `PyRunReport` (`RunReport`) — see the "Strategy builder + run" section and the parity note above; `AtomLift` + `snapshot_signal` / `snapshot_source` bridge candle-rooted Python leaves onto the snapshot-rooted strategy.
- **Metrics** — `PyFill`, `PyTrade`, `PyDrawdownSegment` + one `#[pyfunction]` per `src/metrics.rs` `pub fn`, plus `register_metrics_module`. The submodule is registered on `fugazi.metrics` **and** injected into `sys.modules["fugazi.metrics"]` so `from fugazi.metrics import sharpe` works.
- **Constructors** — leaf sources (`open`, `high`, …, `identity`, `value`), `src_period!` / `bar_period!` / `bar_noarg!` / `bar_period_multi!` macro invocations, hand-written `macd` / `bollinger` / `keltner` / `donchian` / `stoch_rsi`, cross-timeframe (`resample`, `latch`), readiness override (`unstable` — polymorphic over `PyIndicator` / `PySignal`), overlay (`get`).
- **Remote sources** — `SOURCES_RUNTIME`, `parse_interval_token`, `parse_date_token`, `PyBinance`, `PyYahoo`, `PyCoinGecko`, `PyCoinMarketCap`, `fetch`. Each `.candles(...)` returns a DataFrame carrying `time` + OHLCV plus the provider-specific extras threaded through `Atom::overlays` (Binance: `quote_volume` / `n_trades` / `taker_buy_base_volume` / `taker_buy_quote_volume`; Yahoo: `adj_close`). `PyCoinGecko` / `PyCoinMarketCap` are the `OverlaySource` shape instead: `.overlays(...)` returns `time` plus the provider's columns (CoinGecko: `price` / `market_cap` / `total_volume` / `circulating_supply`; CoinMarketCap: `price` / `volume_24h` / `market_cap` / `circulating_supply` / `total_supply`, paid tier) and **no OHLCV**, plus `.ids()` for the ticker/coin-id vocabulary. `set_overlay_columns` is the shared Real/Bool/Str column-dispatch used by both frame builders.
- **`#[pymodule] fn fugazi`** — one `m.add_class` per pyclass, one `reg!` line for every `#[pyfunction]`, then the metrics submodule registration.

Cargo layout: `python/Cargo.toml` depends on `fugazi_core = { package = "fugazi", … default-features = false, features = ["sources"] }` — the wheel gets the async HTTP stack but not the `cli` feature. `pyo3 = "0.29"` with `abi3-py39`.

Build/test the bindings:

- `cargo build` (from `python/` or the workspace root; the top-level `cargo test` also compiles this crate).
- `cargo clippy --all-targets` — keep clean.
- `maturin develop` inside a venv, then `pytest python/tests/` — smoke + README tests. `python/tests/test_readme.py` runs the code blocks in `python/README.md`, so any new binding worth mentioning gets a README example that doubles as a test.
