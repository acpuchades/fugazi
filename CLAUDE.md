# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

`fugazi` is a Rust library (edition 2024, minimal dependencies) of **incremental** technical-analysis primitives. Every primitive owns its internal state and is advanced one sample at a time via `update()`, carrying just enough intermediate state to produce the next output in ~O(1). The same code therefore serves both live streaming and batch backtesting.

The core library's dependency footprint is deliberately small — `serde` + `serde_json` (used in the `sources` module and the CLI's YAML/JSON plumbing), `time` (calendar sources' `Timestamp` decomposition), and `statrs` (standard-normal CDF/inverse-CDF, used by `fugazi::metrics::probabilistic_sharpe` and `deflated_sharpe`). Everything else is gated behind the `sources` and `cli` Cargo features. The prior "zero external dependencies" claim held before the PSR/DSR + calendar-time additions and has been relaxed to "minimal dependencies" as a matter of course; any *new* unconditional dependency is still a judgment call — reach for a stateless closed-form primitive first.

### Bumping the package version

The version is duplicated in **four** places that must stay in lockstep — a `cargo build` catches a Rust/Rust drift but not a Python or README one, so it's worth a checklist:

1. **`Cargo.toml`** — `[package] version = "X.Y.Z"` (the workspace root).
2. **`python/Cargo.toml`** — `[package] version = "X.Y.Z"` (the pyo3 cdylib crate).
3. **`python/pyproject.toml`** — `[project] version = "X.Y.Z"` (the wheel metadata; this is the version `pip install fugazi` sees).
4. **`README.md`** — the `[dependencies]` snippet under `## Install` (`fugazi = "X.Y"` — track the major.minor there, not the patch).

Then `cargo build --workspace` (updates `Cargo.lock`), commit all five files (`Cargo.toml`, `python/Cargo.toml`, `python/pyproject.toml`, `README.md`, `Cargo.lock`), tag `vX.Y.Z`, push both `main` and the tag. The Python README (`python/README.md`) intentionally has no version string — `pip install fugazi` gets the latest, and the wheels tell users which pyo3/Python versions they need.

## Architecture

Three composable layers: indicators (numeric sources), signals (boolean-valued indicators — `Indicator<Output = bool>`), and strategies (the decision layer that trades into a wallet).

### Indicators = the numeric *sources* (`src/indicator.rs`, `src/indicators/`)

`Indicator` has associated `Input`/`Output`, `update(&mut self, Input) -> Option<Output>`, `value()` (the latest output, matching the public `value` field), `is_ready()`, `reset()`, plus warm-up introspection: `warm_up_period()` (**required**, the *exact* number of samples before the first `Some`, accounting for the whole composed chain — wrappers add their window on top of `source.warm_up_period()`, binary carriers take the max of both sides) and `unstable_period()` (default `0`; recursive/IIR smoothers return the extra samples until the seed's residual weight decays below 0.1% — `EmaState`/`WilderState` expose their own `unstable_period()` for this — the same name as the trait method — and wrappers sum it into `source.unstable_period()`; the tolerance is `SETTLE_TOLERANCE = 1e-3` in `smoothing.rs`); provided `stable_period()` = warm-up + unstable. `tests/warm_up.rs` asserts warm-up exactness for the whole catalogue — a new indicator should be added to that battery. Output is `Option` because most indicators need a warm-up (`None` until ready).

The defining design choice: **price-series indicators own their input source** and are generic over it — `Ema<S>`, `Sma<S>`, `Rma<S>`, `Rsi<S>`, `Macd<S>` where `S: Indicator<Output = Real>`, with `Input = S::Input`. So composition is just nesting constructors:

```rust
Ema::new(Current::close(), 20)          // EMA-20 of the close (Input = Candle)
Ema::new(Sma::new(src, 10), 20)         // EMA of an SMA
Rsi::new(Identity::new(), 14)           // RSI of a raw Real stream
```

There is **no pipe/`then`/`Chain`** — chaining *is* construction.

- **Leaf sources** terminate the chain: `Value<I>` (constant that ignores its input), `Identity<I = Real>` (input-typed passthrough — `Identity::new()` still defaults to `Input = Output = Real` for classical scalar chains, `Identity::<Atom>::new()` is the atom passthrough that serves as the *default source* for the candle/calendar leaves below), and the candle accessors under `Current` (`Current::close()`, `Current::volume()`, …; built on the `Field<F, S>` / `CandleField` carrier in `candle.rs`). Every candle- or calendar-input leaf — `Field<F, S>` (i.e. `Close`/`High`/`Low`/`Open`/`Volume`/`Typical`/`Median`), `CurrentBar<S>`, `Calendar<F, S>` (i.e. `Year`/`Month`/`DayOfWeek`/…), `CurrentTime<S>`, `IsWeekday<S>`, `IsWeekend<S>` — is generic over an atom-emitting source `S: Indicator<Output = Atom>` with a default of `Identity<Atom>`, so `Close::new()` still produces the classic `Input = Atom` shape, and `Close::of(some_atom_source)` re-roots it onto any other source that emits `Atom` (a future `!pick` for cross-asset frames, an atom-side `Latch`/`Resample`, etc.). Constructor convention across all six leaves: `T::new()` for the default identity source, `T::of(source)` for a custom one. The type aliases (`Close`, `Year`, `IsWeekday`, …) all carry an `<S = Identity<Atom>>` parameter so `Close::<CustomSource>::of(...)` type-checks.
- **Cross-asset sources** (`types.rs` + `indicators/pick.rs`) let a chain reach into a *multi-asset* input frame. Three core types (all in `types.rs`, exported at the crate root and prelude):
  - **`Frequency`** — the bar cadence enum (`Minute(u32)` / `Hour(u32)` / `Day(u32)` / `Week(u32)` / `Month(u32)`) with a total order **by duration** (not variant tag) and `FromStr` on the canonical `N<unit>` alphabet (`m`/`h`/`d`/`w`/`M`). This is the canonical bar-cadence type in the crate — `cli::calendar::Frequency` is a `pub use` re-export, and `sources::Interval` is its provider-side twin (separate for backward compat with the source layer).
  - **`Selector { symbol: Option<String>, freq: Option<Frequency> }`** — a **partial key** naming *which* asset in a snapshot a `Pick` should read. Both fields default to `None`; the empty selector (`Selector::default()`) is legal and stands for "no query" (the `Pick::new` single-entry-unpack path). `Selector::matches(&self, storage)` uses wildcard semantics — each `None` field on the query matches any storage value, a `Some` field must equal storage — so `Selector::by_symbol("BTC")` matches `{BTC, Some(1h)}` even though the query is silent on `freq`. Constructor shorthands: `by_symbol` / `by_freq` / `exact`.
  - **`Snapshot<K>`** — the keyed collection of atoms itself (a newtype around `HashMap<K, Atom>` with `get`/`insert`/`iter`/`FromIterator`/`From<HashMap>` and — unconditionally — `sole_atom(&self) -> Option<&Atom>` which returns the unique atom in a size-1 snapshot, panics with a diagnostic on 2+, and reads `None` when empty; the specialized `impl Snapshot<Selector>` adds `find(query: &Selector)` for wildcard-aware structural lookup). The CLI/Python layer only ever uses `Snapshot<Selector>`; the generic `K` stays for callers who want to key by their own type in bespoke Rust code.

  Then **`Pick<S = Identity<Snapshot<Selector>>>`** (no `K` generic — `Selector` is baked in) projects one asset out: `Input = S::Input`, `Output = Atom`. Constructors: `Pick::new()` (empty selector — the sole-atom unpack path), `Pick::matching(selector)` (structural match; the empty selector is legal here but dispatches to `sole_atom` too — the empty branch is orthogonal to how the pick was built), `Pick::of(selector, source)` for a custom snapshot-emitting source. Every source-generic leaf composes straight on top: `Close::of(Pick::matching(Selector::by_symbol("BTC")))` is BTC's close as a `Real`-output indicator over `Snapshot<Selector>`, `Close::of(Pick::matching(...)).sub(Close::of(Pick::matching(...)))` is the BTC/ETH spread, `Atr::new(CurrentBar::of(Pick::matching(...)), 14)` is BTC's ATR. A `Pick` with a *non-empty* selector fires `None` when the query matches no entry — the same `None`-until-warm convention every downstream comparison already respects. A `Pick` with an *empty* selector **panics** on a 2+ snapshot rather than silently choosing one — that path is the "single-series strategy accidentally hooked up to multi-asset input" tripwire; the panic surfaces to Python as a `RuntimeError`. **`Atom` equality is by `time`** (two atoms compare equal iff their bar-open `Timestamp`s match, `PartialOrd`/`Ord` sort chronologically with `None` first) — so a snapshot dedups by time and callers can sort a mixed atom slice into a run order without a custom key.

  The Python bindings mirror all of this: `ta.Frequency("1h")`, `ta.Selector(symbol="BTC", freq="1h")`, `ta.Snapshot({...})`, `ta.pick(symbol=..., freq=...)`. Snapshot dict-key coercion is generous: a Python `str` maps to `Selector::by_symbol`, a `Frequency` to `Selector::by_freq`, a `(str, freq)` tuple to a full pair, a `Selector` as-is. `ta.pick()` (no args) is the single-series shortcut and inherits the sole-atom-or-panic semantics. The CLI YAML `!pick { symbol, freq }` tag and the accompanying "root-ish indicators default to `!pick` in specs" plumbing land in a follow-up — that's the last remaining piece for `fugazi-cli`.
- **Position-anchored sources** (`indicators/position.rs`) read the live position rather than the candle stream. A strategy owns one shared `Position` (`Rc<RefCell<PositionState>>`) holding its signed `size`, `entry` price, and the `peak`/`trough` since entry; it updates it from its fills (`apply(side, units, price)`) and each bar (`update(candle)`, folding completed-bars extremes). Its accessors `position.entry()` / `.peak()` / `.trough()` each return a `PositionField` — the field projected into an `Indicator<Output = Real>` (`None` while flat) — so a stop-loss / take-profit **level** is an expression: `position.entry().sub(Atr::new(14).mul(...))` for a fixed stop, `position.peak().mul(Value::new(0.95))` for a trailing one. Only meaningful inside a strategy that drives the `Position`.
- **Bar indicators** consume the whole `Candle` directly (they are not "of a price"): `Atr`, `Adx`, `TrueRange`, and the volume indicators `Obv`, `Vwap`, `Ad` (Chaikin A/D line), `Mfi` (money-flow index). These take only parameters, or none, e.g. `Atr::new(14)`, `Obv::new()`, `Mfi::new(14)`. The cumulative ones (`Obv`/`Vwap`/`Ad`) anchor at construction; `reset()` re-anchors — e.g. at a session boundary for `Vwap`.
- **Two-source indicators**: `Donchian<H, L>` takes a high source and a low source, e.g. `Donchian::new(Current::high(), Current::low(), 20)`.
- **Calendar sources** (`indicators/calendar.rs`) read `atom.time` (an `Option<Timestamp>` on every `Atom`; UTC ms) and decompose it: `Year`, `Month`, `Day`, `Hour`, `Minute`, `Second`, `DayOfWeek` (ISO, 1=Mon..7=Sun), `DayOfYear`, `WeekOfYear` (ISO), `Quarter`, `UnixSeconds`, `UnixMillis` — all built on the `Calendar<F, S> + CalendarField` marker pattern (same shape as `Field<F, S> + CandleField`, and generic over the same atom-emitting source `S` with default `Identity<Atom>`), plus a `CurrentTime` leaf that yields the raw `Timestamp` payload and the two bool signals `IsWeekday` / `IsWeekend`. The CLI's `--series` loader (`src/cli/data.rs`) parses each row's `time` column into a `Timestamp` and calls `Atom::with_time` / `with_overlays_and_time` at load, and every remote provider (`sources/binance.rs`, `sources/yahoo.rs`) already returns `Vec<Atom>` with time populated on every bar — so calendar accessors fire on every CSV-fed or remotely-fetched bar. `atom.time` is only `None` for synthetic in-code atoms (`Atom::new(candle)`) or when the row's `time` label matches no supported shape; every calendar accessor then emits `None` for that bar and the two signals read `false`. Even when `time` is populated, daily-and-higher bars conventionally sit at 00:00 UTC of the session open, so `Hour`/`Minute`/`Second` read as identically `0.0` and carry no information — only day-and-above accessors are meaningful. Anything beyond the two convenience signals is a composition: "Monday" is `!eq { lhs: !day_of_week, rhs: !value 1 }`, "morning trading window" is `!lt { lhs: !hour, rhs: !value 9 }` on intraday data. YAML surface: bare `!year` / `!month` / `!day` / `!hour` / `!minute` / `!second` / `!day_of_week` / `!day_of_year` / `!week_of_year` / `!quarter` / `!unix_seconds` / `!unix_millis` / `!time` (the `Timestamp`-typed leaf) / `!is_weekday` / `!is_weekend`. Python: `ta.year()` … `ta.unix_millis()`, `ta.is_weekday()` / `ta.is_weekend()`; `ta.Atom(candle, time=<ms>)` and `.time` getter. The `time` decomposition uses the `time` crate; the crate's dep list therefore carries `time` unconditionally (see the [Bumping the package version](#bumping-the-package-version) note — versions still stay in lockstep across the four manifests).
- **`Unstable<S>`** (`indicators/unstable.rs`) is a **transparent passthrough** wrapper: `Input = S::Input`, `Output = S::Output`, forwards every method to the inner source *except* `unstable_period()`, which it forces to `0`. `warm_up_period()` is untouched. Built with `Unstable::new(source)`; the fluent `.unstable()` on both `IndicatorExt` and `BoolIndicatorExt` returns it. YAML surface: `!unstable { source: <s> }` (real source) / `!unstable { signal: <s> }` (bool signal). This is the **opt-in override** for the readiness gate — a caller who's happy to trade through a subtree's IIR settling tail wraps it so the strategy's `is_ready()` only waits for the warm-up. It is deliberately *not* the readiness-probe bool signal that the removed `Stable` was: readiness is now a `Strategy` concern, computed on the strategy's own bar count against its consulted `stable_period()`s. See [Safe defaults, opt-in overrides](#safe-defaults-opt-in-overrides) for the design principle.
- `Real = f64` and `Candle` (OHLCV) live in `src/types.rs`.
- Multi-output indicators (`Macd`, `Adx`, `Bollinger`, `Donchian`, `Keltner`, `Aroon`, `Dmi`) expose one named field per output and set `Output` to a small `Copy` struct (`MacdValue`, `AdxValue`, …); single-output ones expose `value: Option<Real>`. Each also has a **component accessor per output** (`macd.line()`/`.signal()`/`.histogram()`, `bands.upper()`/`.middle()`/`.lower()`, `adx.adx()`, `dmi.plus_di()`, …) returning a `Component<Self>` — a single field projected back into an `Indicator<Output = Real>`, so one output of a struct-valued indicator composes and compares like any other source: `macd.line().crosses_above(macd.signal())`, `Current::close().gt(bands.upper())`. Each accessor **clones** the source (one independently-advanced instance per component, like `crosses_above`'s operand clone). The `Component` carrier (`indicators/component.rs`) holds the source plus a `fn(Output) -> Real` selector — one generic carrier, no per-field marker types. **The accessors themselves are generated by the `component_accessors!` macro** in `indicators/component.rs`, invoked once per multi-output indicator (e.g. `component_accessors!(Bollinger<S>, BollingerValue; upper => upper, middle => middle, lower => lower);`) — when adding a new multi-output indicator, invoke the macro; don't hand-write the accessor bodies.
- `StochRsi<S>` is a type alias for `Stochastic<Rsi<S>>` — StochRSI is just the stochastic transform over an RSI source: `Stochastic::new(Rsi::new(src, 14), 14)`.

### Signals = boolean-valued *indicators* (`src/signal.rs`, `src/indicators/{compare,logic,ext}.rs`)

A **signal is just an `Indicator<Output = bool>`** — there is no second trait hierarchy. `Signal` is a thin marker, `trait Signal: Indicator<Input = Candle, Output = bool>` (blanket-impl'd, `?Sized`), naming "a boolean condition over a `Candle`" so a strategy can hold one as `Box<dyn Signal>`. Like any indicator a signal is `None` until warmed; read it as a plain `bool` (false until ready) with `BoolIndicatorExt::is_true` (= `value().unwrap_or(false)`). There is no `src/signals/` module — the comparison/logic/ext pieces live under `indicators/` and are imported from there.

- **Comparisons are bool-output binary ops** over two real sources: aliases `Gt`/`Lt`/`Ge`/`Le`/`Eq`/`Ne` for `Combine<L, R, GtOp>` etc. (`indicators/compare.rs`). The op is **value-carrying** — each holds an absolute `epsilon` (default `DEFAULT_EPSILON = 1e-8`); the fluent `.gt()`/`.lt()`/… builders use the default, `Gt::with_epsilon(a, b, eps)` overrides.
- **Boolean logic** (`indicators/logic.rs`): `And`/`Or`/`Xor` are `Combine<L, R, AndOp>` etc. over two bool sources; `Not` and `Change` are dedicated unary bool-output carriers; `Const<In>` is the constant-bool leaf (the twin of `Value`).
- `IndicatorExt` (blanket over every `Real`-output indicator, in `indicators/ext.rs`) is the fluent builder for **operators only** — comparisons (`gt`/`lt`/`ge`/`le`/`eq_to`/`ne_to`, `above`/`below` — note `eq_to`/`ne_to`, not `eq`/`ne`, to avoid collision with `PartialEq`), arithmetic (`add`/`sub`/`mul`/`div`), lookback (`lag`/`diff`/`ratio`/`roc`), rolling extremum (`rolling_max`/`rolling_min`), the unstable-tail passthrough (`unstable`), and the composed `crosses_above`/`crosses_below`. Named indicators (`Sma`, `Bollinger`, …) are **not** builder methods; construct via their own `::new`. Do not add `.sma()`/`.bollinger()`-style builders.
- `BoolIndicatorExt` (blanket over every `Indicator<Output = bool>`, `?Sized` so it works on `Box<dyn Signal>`) adds the bool view `is_true()` and the combinators `and`/`or`/`xor`/`not` + the single edge primitive `changed` (a `Change` toggle detector) + the unstable-tail passthrough `unstable` (an `Unstable<Self>` that reports `unstable_period() = 0` while forwarding the signal's output).
- **A crossover is not a primitive**: `crosses_above(a,b)` expands to `a.gt(b).and(a.gt(b).changed())` — "comparison is true *and* it just changed". (Clones the operands, so ~2× the source work.)

### Strategies = the decision layer (`src/strategy.rs`)

Unlike the pure layers below it, a strategy **acts**, in three phases: `Strategy` has `update(&mut self, Input)` (advance its signals/indicators — touches only `&mut self`, so updates across strategies are independent and parallelizable), `on_fill(&mut self, &Order<Symbol>)` (a `&mut self` hook the driver calls for each fill the wallet books, so a strategy can track its own position from the fill stream — default no-op), `is_ready(&self) -> bool` (a **safe-by-default** readiness gate — the driver skips `trade()` while it returns `false`, but `update()` and `on_fill()` still run so the warm-up progresses; default is `true`, so a `Strategy` that doesn't care is ready from bar 0), `trade(&self, &mut dyn Wallet<Symbol>)` (read that state and open/adjust/close positions — `&self`, *price-free*; trades against a shared wallet are serial since sizing resolves against its running state), and `reset()` (associated `Input`/`Symbol`). A driver does, each bar: feed the wallet its prices (`update` returns that tick's fills), hand each fill to every strategy's `on_fill`, then `update` each, then `trade` each *only if* `is_ready()` reports `true`. There is deliberately **no one-shot `evaluate`**. Almost every classical single-asset strategy shares one shape — a long/flat/short position driven by a few boolean signals, sized all-in — so the crate ships **`SingleAssetStrategy<Sym>`** (`strategies/single_asset.rs`, `Sym: Clone + PartialEq`): a concrete `Strategy` (Input = Candle) holding four `Box<dyn Signal>` slots (open/close long, open/close short, each defaulting to constant-false). Unconfigured signal slots hold `Const::<Candle>::new(false)` (never fires) and unwired protective levels are `None` (skipped). It tracks a `bars_seen: usize` counter (incremented in `update`, cleared in `reset`) and overrides **`is_ready()`** to compare that count against the largest `stable_period()` across every wired signal (`long`, `close_long`, `short`, `close_short`) **and** every attached protective level — so no trade fires while any consulted source is still on its warm-up or IIR settling tail. A caller who's happy to trade through a subtree's settling tail wraps it in `Unstable` (Rust: `.unstable()`; YAML: `!unstable { source }` / `!unstable { signal }`) so that subtree contributes `0` to the readiness computation. Purely windowed (FIR) chains — SMA crossovers and the like — have `unstable_period() = 0` throughout, so the gate elapses on the last warm-up bar with no explicit override needed. Built with three builders — `long_on(enter, exit)`, `short_on(enter, exit)` (chainable; an opposite-side entry reverses an open position) and the `buy_and_hold(symbol)` constructor. It also carries optional per-side **stop-loss / take-profit levels** (`Box<dyn Indicator<Output = Real>>`) — set via `long_stop_loss`/`long_take_profit`/`short_*`, built against the strategy's own `Position` (exposed by `position()`) — a fixed 5% long stop is `position.entry().mul(Value::new(0.95))`, a 5% trailing long stop is `position.peak().mul(Value::new(0.95))`, an ATR stop is `position.entry().sub(Atr::new(14).mul(Value::new(2.0)))`. It **tracks its own `Position` from fills** (`on_fill` calls `position.apply`, setting the entry at the real fill price and clearing on a flatten) and reads that `Position` for every decision — it never polls the wallet, which is left as the pure execution venue. Its `trade` runs entries first (all-in, reversal-capable), then flatten-to-flat signal exits, then rests the protective levels on the active side. **Entries and signal exits are market orders** (`set`/`close`, followed by `cancel_protective`) — the wallet queues them and fills them a bar later at the next bar's `open` (see the `Wallet`/`PaperWallet` note below), so the bar whose `close` produced the signal is never also its fill bar. **Protective stops are resting orders the wallet owns**: each bar the strategy re-submits the level as a `set_stop` / `set_take_profit` (idempotent, latest-wins, so a trailing level cancel/replaces), and the *wallet* triggers and prices it (fill at the level, or the bar's `open` on a gap). Trailing levels track *completed* bars (`Position::peak`/`trough` exclude the bar in progress), so a resting stop rests one bar after a new extreme and thus fills a bar later than a fixed stop would (fixed stops fill on the same bar they would have before). (This is the generic `(signal, action)` shape earlier designs deliberately avoided — a policy-object/`RuleStrategy`/combined-`evaluate` lineage; it was reintroduced **on request**. `SingleAssetStrategy` is still just "a concrete type implementing the trait", parameterised over its signals — not a rule engine. Don't add policy traits or a `(signal, action)` table beyond it without being asked.) The `src/strategies/` **catalogue of classical strategies** is then a set of **free-function specializations** returning a `SingleAssetStrategy` (`ma_crossover`, `rsi_reversal`, `donchian_breakout`, `keltner_breakout`, … grouped under `trend`/`mean_reversion`/`momentum`/`volume`/`composite`): long/flat = `new(sym).long_on(enter, exit)`, always-in long/short = `new(sym).long_on(up, down).short_on(down, up)`. The one strategy that doesn't fit the long/flat/short, all-in mould — `ZScoreReversion` (reads its `z` indicator directly; long/short with a flat rest) — stays its own bespoke `Strategy` type, and (being outside the fill-tracked shape) still reads the wallet's position via tiny private `is_long`/`is_short` free helpers **at the top of `strategies/mean_reversion.rs`** (its sole caller). Positions size all-in via `value_frac(1.0)`, which survives a reversal (equity, unlike cash), so one `set` reverses and re-sizes all-in exactly — no `enter_all_in` helper.

All in `src/strategy.rs`:
- **`Wallet<Sym>` is a trait** (the portfolio interface taken as `&mut dyn`) — the single **seam** between pure fugazi and a downstream execution system. fugazi stays pure (ships only the in-memory paper impl); a downstream crate implements `Wallet` with a type whose `set_position` *publishes to an event bus / routes to a broker*. The wallet is **priced from outside**: it carries no market view; `update(symbol, candle) -> Vec<Order>` feeds each symbol's bar every tick (fugazi is agnostic to where bars come from) — the `close` marks to market, the `[low, high]` range bounds where a fill may occur, and it **returns the fills booked on that tick** (the fill stream the driver routes to `on_fill`) — and `funds()`/`position(&Sym)`/`price(&Sym)` (the last fed `close`)/`equity()` query it. **Submitting is not filling.** The **market** movements — `set_position` (absolute units, the one required movement), `set` (a `Side` + `Size`, absolute target — opposite side reverses), `close` (flat) — return an `Ack` synchronously: `Ack::Filled(Order)` if a fill happened on the spot, else `Ack::Working(OrderId)` (accepted, fill arrives later in the fill stream). The **resting** protective orders — `set_stop(sym, trigger)` / `set_take_profit(sym, trigger)` (idempotent, latest-wins per symbol; the wallet reads the side from the position, so a stop is the adverse level and a target the favourable one) and `cancel_protective(&sym)` — let a strategy rest a bracket the wallet triggers and prices itself. `PaperWallet` **queues** the market moves and rests the protective legs, filling at the *next* bar's `open` (market) or when a bar trades through the trigger (protective, at the level or the `open` on a gap), so a backtest never fills on the signal's own bar; a live impl fills on the venue's schedule. `WalletError` (`UnknownPrice`, `InvalidPrice`, `PriceOutOfRange`, `InsufficientFunds`) is returned by a live movement that can't be accepted; on `PaperWallet` an infeasible *queued* fill is silently dropped (matching the strategies' `let _ = …`), so its public methods return `Ok`. There is deliberately **no explicit-price primitive on the trait** (the removed `set_position_at`/`close_at`) — fill-pricing lives in the impl, not the strategy — and **no `trade(delta)`** — scale-in is `set_position(position + delta)`. NB: the trading/event-bus/market system itself is **not** in fugazi — it's a separate project that imports fugazi; keep market/IO code out of this crate.
- **Unit-tagged amounts** keep reference currency and instrument units from mixing: `Reference(Real)` (quote/funds currency — `funds`/`equity`/`price`) and `Units<Sym> { symbol, amount }` (signed instrument units — `position`, `set_position`). `Order<Sym>` (`{ symbol, side, units, price, kind, id }`) stays plain `Real` — its `symbol`+`side` imply the unit; it carries the `price` it filled at, an `OrderKind` (`Market`/`Stop`/`TakeProfit`, so the blotter distinguishes a market fill from a trigger fill), and the `OrderId` of the submission it fills (correlating a later fill to its `Ack::Working`). `Order::from_delta(symbol, delta, price, kind, id)` builds the buy/sell for a position change (`None` within `DEFAULT_EPSILON`).
- **`PaperWallet<Sym>`** is the built-in **pure** `Wallet` impl: in-memory `funds` + `HashMap<Sym,Real>` positions + a `HashMap<Sym,Candle>` bar map + a `HashMap<Sym,Pending>` queue of pending market orders + a `HashMap<Sym,Protective>` of resting brackets + a blotter (`Vec<Order>`) + a `next_id` counter. The market `set_position`/`set`/`close` **queue** one move per symbol (latest wins) returning `Ack::Working`; `set_stop`/`set_take_profit` register a resting leg (`Protective { stop, take_profit }`, one bracket per symbol, latest wins); `cancel_protective` drops the bracket. `update(symbol, candle)` marks the bar, flushes the queued market order at the `open`, then matches the resting legs against `[low, high]` (stop-loss precedence; a fill flattens, so the whole bracket is OCO-cancelled — as is a market exit/reversal, via the private `fill_at`). All fills funnel through the private inherent `fill_at` (the old `set_position_at` body, now stamping `kind`/`id` and returning `Result<Option<Order>, WalletError>`); the resting-order fill price is provably in `[low, high]`, so `PriceOutOfRange` is unreachable on the paper path. Caller-owned; adds inherent `new`, `positions()`, `orders()`, `reset()` (full restore to freshly-constructed state; the seed `funds` is retained on the struct) (`equity()` is the trait method, arg-free).
- **`Ack<Sym>`** (`Filled(Order)` | `Working(OrderId)`) is the synchronous submit result; **`OrderId(u64)`** is wallet-minted (a `next_id` counter) so a later fill correlates to its submission. Execution stays **synchronous** — no async/Futures; the per-bar loop is the event loop and live fills arriving between bars reach the strategy via `on_fill`.
- **`Size`** (the magnitude vocabulary): `Units(n)` absolute, `FundsFraction(f)` (= `f·funds/price`, cash), `ValueFraction(f)` (= `f·equity/price`, all-in/target-weight; `1.0` flips cleanly on a reversal), `PositionFraction(f)` (= `f·|position|`, adjust-only). `resolve(price, position, funds, equity) -> magnitude`. Direction comes from `Side` (`Buy`/`Sell`, `.sign()`), not the size.
- There is **no `Market` trait**: the wallet holds its own fed prices, so a multi-asset input just feeds several symbols via `update` and a strategy's `trade` acts on several symbols in one call (multi-asset/pairs in the same type).
- Sizing/direction/short-selling/always-in-market are all just *what the strategy's code does* — no flags. Python (`python/src/lib.rs`) binds `PaperWallet`/`Order`/`Size` (sides as `"buy"`/`"sell"` strings, symbols as `str`; `update` returns the fills list, `set`/`set_position`/`close`/`set_stop`/`set_take_profit`/`cancel_protective`, `Order.kind` as `"market"`/`"stop"`/`"take_profit"`, `WalletError` → `ValueError`); a Python "strategy" is plain Python code driving a `PaperWallet`.

### Run driver = the pure per-bar loop (`src/backtest.rs`)

**`fugazi::backtest::run(&mut strategy, &mut wallet, symbol, candles) -> RunReport<Sym>`** is the pure primitive that walks a `Strategy` over a candle series through any `impl Wallet<Sym>` — it's not backtest-only (a live-`Wallet` impl works too), hence the neutral **`run`** name (not `backtest`). Per bar: iterate the fill stream returned by `wallet.update(symbol, candle)`, routing each fill to `strategy.on_fill` and appending it (bar-tagged) to the report's blotter; then `strategy.update(candle)`; then `strategy.trade(wallet)` **only if `strategy.is_ready()` reports true** (the safe-by-default readiness gate — `is_ready` on the trait defaults to `true`, so a strategy that doesn't opt in trades from bar 0 as before); then push `wallet.equity().0` to the equity curve. The report is **`RunReport<Sym> { equity_curve: Vec<Real>, fills: Vec<Fill<Sym>>, initial_equity: Real }`** — `equity_curve` has one mark-to-market entry per input bar, `fills` records every booked `Order<Sym>` (each in a `Fill<Sym> { bar: usize, order: Order<Sym> }`) in the order the wallet produced them, and `initial_equity` is `wallet.equity()` captured **before** the first bar (the seed value `total_return` / `cagr` compound against; for a freshly `PaperWallet::new(cash)` it equals `cash`). `S::Symbol: Clone` (the symbol is cloned per bar to feed `wallet.update`); no bound on `Sym` is imposed for anything else. This is the loop the CLI's `run()`/`evaluate()` (`src/cli/backtest.rs`) sit on — the CSV writing, plotting, and console reporting all happen in a **post-run** pass over `report.fills` / `report.equity_curve`, after `fugazi::backtest::run` has returned. **`Fill` and `RunReport` are re-exported at the crate root** (`fugazi::Fill`, `fugazi::RunReport`); the driver stays namespaced (`fugazi::backtest::run`) so the top-level `run` name isn't monopolized. A caller that needs trade-streaming callbacks, per-bar hooks, or its own bar timing writes the loop directly instead — the primitive is deliberately opinion-free.

### Metrics = one function per metric (`src/metrics.rs`)

**There is no aggregate `compute`.** Every metric is its own `pub fn` in `fugazi::metrics`, so a caller picks whichever numbers matter and calls those; a caller that wants all of them calls all of them. Metrics that share an expensive intermediate consume public data types — three **intermediate builders** turn `RunReport` artefacts into what each metric family needs:
- **`per_bar_returns(equity: &[Real], initial_equity: Real) -> Vec<Real>`** — `(equity[i] - prev) / prev`, seeded from `initial_equity`; consumed by every return-moment / risk-adjusted metric.
- **`reconstruct_trades<Sym>(fills: &[Fill<Sym>]) -> Vec<Trade>`** — walks the blotter with one signed position and a volume-weighted entry; opposite-side fills close (or reverse); one closed leg = one `Trade { entry_bar, exit_bar, side, units, entry_price, exit_price, pnl, return_ratio }` (`bars_held()` = `exit_bar - entry_bar`). Consumed by every trade-level metric.
- **`drawdown_segments(equity: &[Real]) -> Vec<DrawdownSegment>`** — one peak → trough → recovery-or-end segment per drop; `DrawdownSegment { peak_bar, trough_bar, depth_ratio, duration_bars, underwater_bars }` — `underwater_bars` counts bars strictly below the peak so a caller sums them for `time_in_drawdown_ratio` without re-walking.

The metric catalogue itself is the standard set — return moments (`total_return`, `cagr`, `mean_return`/`median_return`/`stddev_return`/`best_return`/`worst_return`, `positive_bars_ratio`, `skewness`, `kurtosis`, `value_at_risk`/`conditional_value_at_risk` at any confidence, `tail_ratio`, `annualized_return`/`annualized_volatility`), risk-adjusted (`sharpe`, `sortino`, `calmar`, `omega`, `ulcer_index`, `ulcer_performance_index`), higher-moment / multiple-testing Sharpe corrections (`probabilistic_sharpe` — Bailey/LdP 2012 PSR against a caller-supplied benchmark Sharpe; `deflated_sharpe` — Bailey/LdP 2014 DSR against the expected max-SR under a normal null across `n_trials` grid candidates; each also has a `*_from_stats` variant that takes pre-aggregated Sharpe/skew/kurtosis so the `optimize` CSV writer can compute DSR per row from a `Metrics` document without re-scanning returns), drawdown (`max_drawdown`, `max_drawdown_duration`, `average_drawdown`, `average_drawdown_duration`, `drawdown_count`, `time_in_drawdown_ratio`, `recovery_factor`), and trade-level (`total_trades`, `winning_trades`/`losing_trades`/`flat_trades`/`long_trades`/`short_trades`, `max_consecutive_wins`/`max_consecutive_losses`, `win_rate`, `profit_factor`, `payoff_ratio`, `expectancy`, `kelly_fraction`, `average_win`/`average_loss`/`largest_win`/`largest_loss`, `average_trade_return`, `average_bars_held`/`min_bars_held`/`max_bars_held`, `exposure_ratio`). **Values are in natural units** (`0.15` = +15% total return, not 15.0) — the presentation layer scales to percent. Ratios whose denominator can vanish return `Option<Real>` and read `None` on the degenerate case (zero variance for Sharpe, no losing trade for a profit factor, non-positive endpoints for CAGR, …); metrics that are always well-defined return `Real` and read `0.0` on empty input. The PSR/DSR pair uses `statrs` for the standard-normal CDF (Φ) and its inverse (Φ⁻¹) — the one place a closed-form primitive wasn't practical without vendoring an Abramowitz-Stegun approximation.

**The library core stays lean** — no serde on `Metrics`-anything (there is no `Metrics` struct in the library at all); unconditional deps are limited to the handful listed at the top of this file. The optional dependencies come in behind two default-on Cargo features: **`sources`** pulls `reqwest`/`tokio`/`time`/`thiserror` for the remote-candle-provider layer (`fugazi::sources`, `fugazi::TimedCandle`), and **`cli`** (which implies `sources`) pulls everything only the binary uses — `clap`/`clap_complete`/`serde_norway`/`csv`/`anyhow`/`indicatif`/`console`/`rayon`/`interim`. All CLI deps are pure Rust — no system libs link in — so the manylinux/wheel build path is uniform: a caller who only wants the pure indicator/backtest/metrics core still builds with `default-features = false` (empty-ish dep tree), and the Python wheel keeps `default-features = false, features = ["sources"]` for the async HTTP stack alone. The `fugazi` binary has `required-features = ["cli"]`, so it just builds by default without a special toggle. **No plotting.** The CLI intentionally emits data files only (`trades.csv`, `returns.csv`, `metrics.yml`, and — under `-w LEN` — `metrics.csv` + `rolling.csv`); charts are a downstream concern (R/Python/notebook — see the README's analysis-workflow section). The CLI's aggregate `Metrics` document (`src/cli/metrics.rs`) is what carries the serde derives + `skip_serializing_if` attrs + the YAML field names (`sharpe`, `max_pct`, `annualized_mean_pct`, …), and is populated field-by-field by a thin **`metrics::from_report<Sym>(&RunReport<Sym>, bars_per_year, risk_free_rate) -> Metrics`** glue that calls one library function per field (scaling to `_pct` at the assignment site). Everything downstream of that struct — `write_yaml`, `resolve_metric` (serde_json name-based lookup for the `optimize` subcommand's `--best-by` / `--metrics` columns) and its underlying **`MetricKey`** handle (a validated-once dotted path plus a `resolve(&Metrics) -> Result<Option<Real>>` method; construct via `MetricKey::from_name(name, sample)` and reuse across many rows to avoid re-walking the JSON tree per lookup), `report_slice` (the sub-run over a bar range: equity slice, rebased fills, prior-bar initial equity — the shared measurement primitive), `windowed_from_report` and `rolling_from_report` (the twin reductions: non-overlapping N-bar windows and rolling stride-1 N-bar windows respectively; `-w/--windowed LEN` on `run` emits both to `metrics.csv` and `rolling.csv` via `flatten`, whose fixed dotted-name column list a test keeps in lock-step with the serialized document — `metrics.yml` is still always written alongside as the whole-run summary; on `optimize`, `-w` still uses only the non-overlapping variant, where each `-m` metric becomes `_mean`/`_std` columns via `mean_std` and `--best-by` ranks by the windowed mean, shifted against the row by `-k/--risk-aversion` stddevs — direction-aware: `−k·std` for higher-is-better metrics, `+k·std` for lower-is-better — because that ranker needs *independent* samples to interpret the stddev as a confidence adjustment; the rolling variant is for plotting a continuous curve, not for cross-window aggregation. `optimize` also gets a trailing **`deflated_sharpe` column** — the per-row DSR against the grid-wide null, where `N` = number of trials and `Var[SR]` is the sample variance of the grid's annualized Sharpes (computed once in `compute_dsr_context` on the collected rows). In whole-run mode each row's summary is its own `metrics.risk_adjusted.sharpe`; in windowed mode each row's summary is the cross-window arithmetic mean of window Sharpes, with skewness / kurtosis aggregated the same way and `n_returns` the sum of window bar counts (an aggregation choice shared with the existing `_mean` columns — the mean-of-window skew isn't quite the pooled-returns skew, but it keeps the DSR cell comparable to its neighbours). Emitted for both modes because windowing regularizes but does not eliminate the "we searched N cells and picked the winner" selection bias — a mean-Sharpe max is still a max-of-many. The column is omitted only when fewer than two rows have a defined summary Sharpe or the trial variance is zero), and the shared measurement primitive (`report_slice`) — stays in the CLI. `run` and `optimize` measure over the whole run; the metrics stack has no crop of its own. The strategy layer's readiness gate is what holds the first trade until every consulted source is past its `stable_period()` — so the equity curve is flat during warm-up and non-flat once the strategy trades, and `total_return` / `cagr` / `sharpe` etc. reflect that shape. (`get`'s `--keep-unstable` flag still exists — same pattern one level up: it opts out of the per-source stable-period trim applied when writing an overlay CSV. See [Safe defaults, opt-in overrides](#safe-defaults-opt-in-overrides).) The library's `Trade` and `DrawdownSegment` structs are re-exported at the crate root (`fugazi::Trade`, `fugazi::DrawdownSegment`); the metric functions themselves stay namespaced (`fugazi::metrics::sharpe(&returns, rf, bpy)`, etc.).

### Generic transform ops (`src/indicators/ops.rs`)

Source-wrapping carriers, each driven by an operator type so a new operator is a trait impl, not a new type:
- `Combine<L, R, Op>` (binary, `BinaryOp`): **one carrier for all binary ops**, generic over the op's input/output via associated `Lhs`/`Rhs`/`Output` and holding the op **by value**. Serves arithmetic `Add`/`Sub`/`Mul`/`Div` (`Real,Real→Real`; `Div` → `None` on /0), the comparisons in `indicators/compare.rs` (`Real,Real→bool`, the op carrying its epsilon) and the boolean logic in `indicators/logic.rs` (`bool,bool→bool`). `Combine::new` needs `Op: Default`; comparison ops also get `Combine::with_epsilon`.
- `Lookback<I, Op>` (unary, relates a source to its value `period` steps ago, `LookbackOp`, zero-sized markers): `Lag` (past value), `Diff` (`x[t]-x[t-n]`), `Ratio` (`x[t]/x[t-n]`), `Roc`.
- `Extreme<S, Op>` (rolling extremum, `ExtremeOp` = `MaxOp`/`MinOp`): `RollingMax`/`RollingMin`.

### Shared cores (`pub(crate)`)

Bare `Real -> Real` math with **no source and no `Indicator` impl**, so both source-wrapping indicators and indicators smoothing values they compute *internally* share one implementation:
- `smoothing.rs`: `EmaState` (EMA recurrence) and `WilderState` (Wilder/RMA, mean-seed). `Ema`/`Macd` use `EmaState`; `Rma` uses `WilderState`; `Rsi` uses two (gain/loss); `Atr` = `TrueRange` + `WilderState`; `Adx` uses four.
- `stats.rs`: `WindowStats` (windowed sum + sum-of-squares → `mean`/`variance`/`stddev`) backs `Sma`/`StdDev`/`Bollinger`; `WindowExtreme<Op>` (monotonic-deque rolling extremum) backs `Extreme`/`RollingMax`/`RollingMin` and `Stochastic`.

## Safe defaults, opt-in overrides

Numbers this crate produces during a source's warm-up or IIR settling tail are *unsettled*: they exist, but they depend on the seed, on where the window happens to start, or on both. Every knob that could paper over an unsettled bar therefore biases toward **waiting**, with a single-name **flag / wrapper / period** for the caller who's considered the tradeoff and wants to trade through it. Current instances of the pattern:

- **Strategy readiness.** `Strategy::is_ready()` (default `true`) gates `trade()` in `fugazi::backtest::run`; `SingleAssetStrategy` overrides it to `bars_seen >= max(stable_period across every wired signal + attached level)`. Opt-out: `Unstable<S>` (Rust `.unstable()`, YAML `!unstable { source | signal }`, Python `.unstable()` / `ta.unstable(x)`) is a transparent passthrough that reports `unstable_period() = 0` for the wrapped subtree.
- **`fugazi get` overlays.** The CLI trims each overlay column's pre-`stable_period()` cells before writing the CSV. Opt-out: `--keep-unstable`, which emits every sample from bar 1.
- **`-w/--windowed` duration form.** `-w 1d`/`-w 1w`/… resolves against the trading calendar (see `WindowSpec::resolve`), and demands both an explicit `AssetClass` (`--stocks`/`--forex`/`--crypto`) and a resolvable bar cadence — a silent Stocks default would change a window's bar count in a way that's hard to notice on 24/7 data, and a missing cadence has no defensible fallback. Opt-out: the plain bar-count form `-w N`, which needs neither.
- **Explicit periods.** Every windowed constructor takes an explicit `period` (asserted `> 0`); risk-adjusted metrics take an explicit risk-free rate and bars-per-year. No "sensible defaults" that would hide the choice.

When adding a new knob that touches unsettled data: pick the value that's safest when the user forgets to think about it, and provide *one* mechanism to opt out. Don't add a wrapper whose default flips the polarity — the point of the pattern is that reading a spec bottom-to-top tells you what a distracted operator gets, and the exceptions are named right there in the tree.

## Conventions and gotchas

- **Composition is construction.** A new "X of Y" indicator takes its source `S` in `new` (or `of`, for source-generic leaves whose zero-arg `new()` uses the default `Identity<Atom>` root — see the [Leaf sources](#indicators--the-numeric-sources-srcindicatorrs-srcindicators) bullet) — with the right output constraint (`S: Indicator<Output = Real>` for numeric wrappers like `Ema`/`Sma`, `S: Indicator<Output = Candle>` for bar indicators, `S: Indicator<Output = Atom>` for the atom-input leaves); don't add pipe combinators.
- **Use the cores, not each other's public types.** Internal smoothing of computed scalars uses `EmaState`/`WilderState` (Real recurrence). The public `Rma<S>`/`Ema<S>` wrap a *source* and can't smooth values you computed inline.
- **Adding an operator** (arithmetic/comparison/boolean/lookback): add an `*Op` type implementing the relevant `*Op` trait plus a type alias — never a macro. Arithmetic/boolean/lookback ops are zero-sized `Default` markers; comparison ops are value structs carrying `epsilon`. They live by their carrier — `indicators/{compare,logic}.rs` for the `Combine`-based ones, `indicators/ops.rs` for `Lookback`/`Extreme`.
- `Combine` (arithmetic, comparisons, and `And`/`Or`/`Xor`) feeds the *same* input to both sides, so it requires `Input: Clone`. Use `lhs`/`rhs` naming for binary operands.
- `Combine` holds its op **by value** (so a comparison can carry epsilon); `Lookback`/`Extreme` hold a zero-sized op as `PhantomData<fn() -> Op>`. Input-ignoring / field-marker leaves (`Value<I>`, `Const<I>`, the `F` in `Field<F, S>` / `Calendar<F, S>`) use `PhantomData<fn(I)>` / `fn() -> F` to satisfy the constraint rules (avoids E0207); `Identity<I>` similarly uses `PhantomData<fn(I) -> I>` for the pinned input=output relationship.
- `Change` is a **bidirectional** toggle detector (fires on any transition); directional events come from pairing it with a comparison (see `crosses_above`).
- Constructors `assert!(period > 0, ...)`; document warm-up length in the type's doc comment, and implement `warm_up_period()` to match it exactly (plus `unstable_period()` when the indicator smooths recursively).
- A comparison/edge is **`None` until** every source it depends on is warmed up (it reads `false` via `.is_true()`); a boolean op (`And`/`Or`/…) is likewise `None` until both sources are ready, so an edge that would coincide with warm-up is not detected (no spurious first-bar trade).

## CLI internals (`src/cli/`)

The CLI is one binary (`fugazi`) with several subcommands; layout by concern:

- **`main.rs`** — clap definitions, subcommand dispatch. Owns the `mod` declarations and the top-level `Cli` type.
- **`run.rs`, `optimize.rs`, `backtest.rs`** — the two user-facing subcommand drivers (`run`, `optimize`) sit on the pure `backtest` module (`run_iteration`, `evaluate`, `evaluate_windowed`). `backtest.rs` deliberately owns no IO; `run.rs` / `optimize.rs` do the CSV writing, console output, and file paths.
- **`get.rs`** — the `fugazi get` subcommand (fetch OHLCV from remote providers). Its private helpers include `write_candles_csv` (formerly `write_csv`) and `parse_spec` for `provider:symbol[freq]` grammar.
- **`spec/`** (directory) — YAML-deserializable mirror of the fugazi composition API, one file per layer:
  - `spec/source.rs` — `SourceSpec` (real-valued source enum + `default_source`/`default_high`/`default_low`/`default_bar_source` helpers, all `pub(super)`).
  - `spec/signal.rs` — `SignalSpec` (boolean condition enum + `eps` tolerance helper).
  - `spec/strategy.rs` — `SideSpec`, `StrategySpec`, `DynSingleStrategy` (the built strategy wrapper).
  - `spec/mod.rs` — module doc, `pub use` re-exports, and the tests for all three layers.
- **`costs/`** (directory) — `--costs` parsing and resolution:
  - `costs/spec.rs` — CLI-arg parsing into `CostSpec` (a term list); `CostTerm` variants + `split_top_commas` / `parse_term`.
  - `costs/config.rs` — everything downstream: normalization, `CostConfig`, `LegConfig<T>`, `ScopedEntry<T>`, the typed `CommissionSpec`/`SpreadSpec`/`SlippageSpec` model enums, and `impl CostConfig::resolve`.
  - `costs/mod.rs` — module doc, `pub use` re-exports, tests.
- **`dyn_indicator.rs`** — the **runtime-typed indicator handle** the YAML builder produces. `DynIndicator` trait + `DynValue` payload enum (`Real | Bool | Atom | Candle`) + `DynType` tag. The `Adapter` wraps concrete library indicators as `DynIndicator`s; `AsReal` / `AsBool` / `AsCandle` cross back the other way so a boxed handle drops into a library constructor. **Any new YAML-visible indicator plugs in here.**
- **`csv_source.rs`** (formerly `file.rs`) — the local CSV candle source for `fugazi get csv:PATH`; sibling to `sources/binance.rs` and `sources/yahoo.rs` conceptually, but CLI-side because it owns the CSV parsing.
- **`data.rs`** — the `--series` data frame builder (multiple `@file.csv` + inline literal specs, full-joined on `symbol` + `time`). Uses `crate::csv_source::detect_delimiter` (the shared CSV delimiter probe).
- **`overlay.rs`** — `--overlay` parsing for `fugazi get`. Its scope type (`OverlayScope`) uses `sources::Interval` (not `Frequency`) because a get scope binds to a remote-fetch interval, so it stays its own type — but the bracket-splitting delegates to `calendar::parse_scope_parts` (see below).
- **`calendar.rs`** — asset-class + bar-frequency vocabulary. Owns `Frequency`, `AssetClass` (with `trading_days_per_year` / `trading_hours_per_day` / `trading_seconds_per_bar`), `Scope { symbol, freq }`, `ScopedFrequency` (formerly `FrequencySpec`), and the shared **`parse_scope`** + **`parse_scope_parts`** helpers. Also owns **`WindowSpec`** (the `-w/--windowed` grammar: `Bars(NonZeroUsize) | Duration(Frequency)` — a plain count or a time suffix in the `m/h/d/w/M` alphabet, resolved via `WindowSpec::resolve(bar_freq, class)` as `win.trading_seconds / bar_freq.trading_seconds`; the duration form errors on missing `class` or `bar_freq` — see [Safe defaults, opt-in overrides](#safe-defaults-opt-in-overrides)), **`parse_time_to_millis`** (RFC3339 / `YYYY-MM-DD [HH:MM:SS]` / integer epoch s or ms → UTC ms — consumed by `data.rs` at load to populate `Atom::time`), and **`detect_frequency_from_atoms`** (reads `Atom::time` directly, no re-parse).
- **`metrics.rs`** — the CLI's aggregate `Metrics` document + `MetricKey` handle + `resolve_metric` wrapper. Details in the metrics section above.
- **`input.rs`** — the `@file`-or-inline `Source` type used by every text-taking flag (`--params`, `--overlay`, `--costs`, the strategy positional).
- **`params.rs`, `convert.rs`, `list.rs`, `completions.rs`, `pool.rs`, `style.rs`** — auxiliary: `!param` substitution, YAML→JSON normalization, `fugazi list`, shell completions, rayon pool builder, terminal styling. None are entry points; touch on demand.

## Python bindings (`python/src/lib.rs`)

The Python surface is a **type-erased mirror** of the Rust library, packaged as one pyo3 cdylib (`fugazi-python` crate → `fugazi` importable module). The core library is compile-time-monomorphised over its source generics (`Ema<S>` etc.); Python can't carry those generics across the FFI boundary, so the whole file is one big erase-then-dispatch: every `I -> Real` indicator becomes a `Source<I>` (boxed `DynIndicator`), every `I`-input bool becomes a `SignalBox<I>`, multi-output ones become a `MultiBox<I>`. The `AnySource` / `AnySignal` / `AnyMulti` enums record which of the two input domains — candle-rooted (`Input = Atom`) or value-rooted (`Input = Real`) — an erased chain lives in, and the `map_source!` / `combine_sources!` / `source_to_signal!` / `sources_to_signal!` / `map_signal!` / `combine_signals!` / `map_multi!` / `combine_multi!` macros dispatch on those tags. **The rule is: a Python constructor that mirrors a Rust one uses those macros; it does not name a concrete `Ema<Sma<Current, …>, …>` type.**

### Parity discipline: keep Python in sync when the library grows

The Python bindings ship the same primitives as the Rust core, with three deliberate omissions (see below). **When a Rust API is added, extended, or renamed, mirror it in `python/src/lib.rs` in the same PR** — Python drift is otherwise silent (the Rust build still passes) and only surfaces when a user reports missing bindings.

The mirror rules by layer:

- **New indicator / signal / operator on the pure layer** → add a `#[pyfunction]` in `python/src/lib.rs`, register it in the `#[pymodule] fn fugazi` reg list, and add a smoke test in `python/tests/test_fugazi.py`. Single-output real-source constructors use `src_period!`; bar-only constructors use `bar_period!`/`bar_noarg!`; multi-output ones use `bar_period_multi!` or a hand-written `map_multi!`/`combine_multi!` body. If it's a new fluent method on `IndicatorExt` / `BoolIndicatorExt`, add a `#[pymethods]` entry on `PyIndicator` / `PySignal` alongside the existing ones (and update the `__add__`/`__and__` etc. dunders if it composes with operators).
- **New metric function in `src/metrics.rs`** → add a `#[pyfunction]` in the metrics section, add its name to the `register_metrics_module` reg list. Signature is mechanical: `Vec<Real>` for slice inputs, `Vec<PyTrade>` / `Vec<PyDrawdownSegment>` / `Vec<PyFill>` for the intermediate types (converted to the native `Vec<Trade>` etc. inside the body). Ratios returning `Option<Real>` stay `Option<Real>` on the Python side (materialises as `None`); metrics returning `Real` stay `f64`; bar counts stay `usize`.
- **New field on `Trade` / `DrawdownSegment` / `Order` / etc.** → add a `#[getter]` on the corresponding `PyTrade` / `PyDrawdownSegment` / `PyOrder` and update its `__repr__`.
- **New remote provider under `src/sources/`** → add a `Py*` client class mirroring `PyBinance` / `PyYahoo`, register it, and add a branch in the `fetch(provider=…)` dispatcher.
- **New leaf source / component accessor** → add a `#[pyfunction]` using `leaf!` or a `candle_source(...)` body, register it.
- **Changes to `Candle` / `Atom` / `OverlayInfo` / `Schema` / `SchemaBuilder`** → update `PyCandle` / `PyAtom` / `PyOverlayInfo` / `PySchema` / `PySchemaBuilder` field-for-field.

**Intentionally not bound** (do not mirror without being asked):

- The `Strategy` trait, `SingleAssetStrategy`, and the `src/strategies/` catalogue. Python is imperative: a Python "strategy" is a plain-Python loop that reads indicators/signals and drives a `PaperWallet`. Re-shipping the 19 Rust recipes as Python constructors would double the surface without matching how Python users actually assemble strategies.
- `fugazi::backtest::run`, `RunReport`, and the whole `run_iteration` / `evaluate*` pipeline. No Python-facing strategy handle → no primitive to hand to `run`. The bar loop is user-owned Python.
- The CLI (`src/cli/`). It's a separate binary; no Python analogue is intended.
- The `Wallet` trait itself. Python only exposes `PaperWallet` — a live wallet would live in a downstream Rust crate implementing `Wallet` and be re-bound separately.
- Rust-internal types that only make sense inside the strategy layer (`Position`, `PositionField`, `Ack`, `OrderId`, `Reference`, `Units`). Python takes `float`s for prices and unit counts and returns `PyOrder` (with `bar` tagging done via `PyFill` when the user drives a metrics workflow).

Layout inside `python/src/lib.rs` (single file, ~3400 lines — grep by section header):

- **Type-erasing carriers** — `DynIndicator` / `DynSignal` / `DynMulti` traits, `Source` / `SignalBox` / `MultiBox` boxes, the `MultiOutput` shim, `ResampleThen`.
- **Input domain** — `AnySource` / `AnySignal` / `AnyMulti` enums, `pair()` resolver, `domain_mismatch()`, and the `map_*` / `combine_*` macros. **All new binding bodies use these macros.**
- **Python classes** — `PyCandle`, `PySchema`, `PySchemaBuilder`, `PyOverlayInfo`, `PyAtom`, `PyIndicator`, `PySignal`, `PyMulti`, `PyWallet`, `PyOrder`, `PySize`.
- **Strategy layer** — `PyWallet` with `set` / `set_position` / `close` / `set_stop` / `set_take_profit` / `cancel_protective` / `update` (returns `Vec<PyOrder>` — a bar-loop that later hands orders to metrics wraps them in `PyFill(bar, order)`).
- **Metrics** — `PyFill`, `PyTrade`, `PyDrawdownSegment` + one `#[pyfunction]` per `src/metrics.rs` `pub fn`, plus `register_metrics_module`. The submodule is registered on `fugazi.metrics` **and** injected into `sys.modules["fugazi.metrics"]` so `from fugazi.metrics import sharpe` works.
- **Constructors** — leaf sources (`open`, `high`, …, `identity`, `value`), `src_period!` / `bar_period!` / `bar_noarg!` / `bar_period_multi!` macro invocations, hand-written `macd` / `bollinger` / `keltner` / `donchian` / `stoch_rsi`, cross-timeframe (`resample`, `latch`), readiness override (`unstable` — polymorphic over `PyIndicator` / `PySignal`), overlay (`get`).
- **Remote candle sources** — `SOURCES_RUNTIME`, `parse_interval_token`, `parse_date_token`, `PyBinance`, `PyYahoo`, `fetch`. Each `.candles(...)` returns a DataFrame carrying `time` + OHLCV plus the provider-specific extras threaded through `Atom::overlays` (Binance: `quote_volume` / `n_trades` / `taker_buy_base_volume` / `taker_buy_quote_volume`; Yahoo: `adj_close`).
- **`#[pymodule] fn fugazi`** — one `m.add_class` per pyclass, one `reg!` line for every `#[pyfunction]`, then the metrics submodule registration.

Cargo layout: `python/Cargo.toml` depends on `fugazi_core = { package = "fugazi", … default-features = false, features = ["sources"] }` — the wheel gets the async HTTP stack but not the `cli` feature. `pyo3 = "0.29"` with `abi3-py39`.

Build/test the bindings:

- `cargo build` (from `python/` or the workspace root; the top-level `cargo test` also compiles this crate).
- `cargo clippy --all-targets` — keep clean.
- `maturin develop` inside a venv, then `pytest python/tests/` — smoke + README tests. `python/tests/test_readme.py` runs the code blocks in `python/README.md`, so any new binding worth mentioning gets a README example that doubles as a test.
